CH 01

Reliability, Scalability & MaintainabilityThe three properties every system must be designed around

Reliability fault-tolerant

The system continues working correctly even when things go wrong. A fault is one component deviating; a failure is the whole system going down. Good systems tolerate faults before they become failures.

  • Hardware faults: disk dies, RAM flips a bit → use RAID, redundant power, multi-AZ replicas
  • Software faults: a bug that kills every node when a particular input arrives → chaos testing, careful error handling
  • Human error: config mistakes cause more outages than hardware → sandbox envs, gradual rollouts, fast rollback

Scalability load & performance

As load grows, can the system cope? You must first describe load precisely with load parameters, then measure performance under load.

  • Twitter fan-out example: 12 writes/s for tweets but 345K reads/s for home timelines. The read/write ratio shapes the entire architecture.
  • Throughput (batch: records/sec) vs latency (online: p99 response time)
  • p99 matters more than median — your slowest users drive infrastructure decisions

Maintainability operability · simplicity · evolvability

  • Operability: easy for ops teams to run — good monitoring, runbooks, config management
  • Simplicity: reduce accidental complexity. A system with 8 implicit state machines is harder to maintain than one with 1 explicit one
  • Evolvability: easy to change. Agile, TDD, rolling deploys. The system should accommodate new requirements without rewrites

Likely interview questions

  • How would you design a system to handle 10x traffic growth with no downtime?
  • What's the difference between latency percentiles — why do engineers care about p99 not mean?
  • Walk me through how you'd make a service fault-tolerant at the infrastructure level.

CH 03

Storage & RetrievalHow databases actually store and find data — the two fundamental structures

LSM-Trees (Log-Structured Merge) write-optimised

Used by: Cassandra, RocksDB, LevelDB, HBase

  • Writes go to an in-memory memtable, then flushed to immutable SSTables on disk
  • Background compaction merges SSTables, reclaims space, keeps read paths short
  • Pros: very fast sequential writes, high write throughput
  • Cons: reads may scan multiple SSTables; compaction uses I/O bandwidth; bloom filters help
Write → memtable → flush → SSTable L0 ↓ compaction SSTable L1 → L2 → ... Read → check memtable → L0 → L1 → L2 (bloom filter skips most)

B-Trees read-optimised

Used by: PostgreSQL, MySQL (InnoDB), SQLite, Oracle

  • Data stored in fixed-size pages (typically 4 KB), organised as a balanced tree
  • Each write updates the page in place → requires a Write-Ahead Log (WAL) for crash recovery
  • Pros: predictable O(log n) reads, good for range queries, mature and well-understood
  • Cons: random writes (expensive on spinning disks), write amplification, fragmentation over time
Root page ├── Internal page (keys + child pointers) │ ├── Leaf page [k1, v1] [k2, v2] [k3, v3] │ └── Leaf page [k4, v4] [k5, v5] └── Internal page └── Leaf page [k6, v6]

Indexes — Secondary, Clustered, Covering query performance

  • Clustered index: data stored in index order (InnoDB primary key). Fast range scans, only one per table
  • Secondary index: points back to primary key or heap row. Extra lookup needed (unless covering)
  • Covering index: index contains all columns needed by the query — no heap lookup ("index-only scan")
  • Composite index: (last_name, first_name) — column order matters. Left-prefix rule applies
  • Multi-value / inverted index: used in full-text search (Elasticsearch) — maps term → list of document IDs

OLTP vs OLAP workload type

  • OLTP: many short reads/writes, user-facing, latency-sensitive → row-oriented storage
  • OLAP: few long analytical queries over millions of rows → column-oriented storage
  • Column storage compresses well (run-length, bitmap encoding) and only reads relevant columns
  • Example: Redshift, BigQuery, Snowflake all use columnar storage

Data Warehouses & Stars analytics pattern

  • Star schema: central fact table (events/transactions) + dimension tables (users, products, dates)
  • Snowflake schema: normalised dimensions (dimensions broken into sub-tables)
  • Fact tables can have billions of rows; queries aggregate across time ranges
  • ETL pipeline moves data from OLTP → data warehouse nightly (or in real-time with CDC)

Likely interview questions

  • Compare LSM-trees vs B-trees. When would you choose Cassandra over Postgres for storage?
  • What is write amplification and why does it matter?
  • How does a column-oriented database make analytical queries faster?
  • What's the difference between a covering index and a secondary index?

CH 05

ReplicationKeeping copies of data on multiple machines — one of the hardest problems in distributed systems

Single-Leader Replication most common

All writes go to the leader. Leader sends a replication log to followers. Reads can go to followers (eventual consistency) or leader (linearizable).

  • Synchronous follower: leader waits for follower ack before confirming write → durable but slow
  • Asynchronous follower: leader confirms immediately → fast but follower can lag
  • Semi-synchronous: one synchronous follower + rest async → balance
  • Replication lag problems: read-your-own-writes, monotonic reads, consistent prefix reads
Client Write ↓ [Leader] → WAL/binlog → [Follower 1] (sync) → [Follower 2] (async, may lag) → [Follower 3] (async, may lag)

Multi-Leader Replication multi-datacenter

Multiple nodes accept writes. Used when you have multiple datacenters and need local writes in each.

  • Conflict resolution is the core problem: two leaders accept conflicting writes to the same record
  • Strategies: last-write-wins (LWW, loses data), merge (CRDTs), custom logic, or avoid conflicts entirely
  • Real example: Google Docs uses operational transformation to merge concurrent edits
  • Beware: autopilot conflict resolution with LWW silently drops writes

Leaderless Replication (Dynamo-style) cassandra / dynamo

Any replica can accept writes. Reads and writes use quorums.

  • Quorum: w + r > n ensures overlap. With n=3: w=2, r=2 guarantees reading at least one up-to-date node
  • Read repair: reader detects stale replica and updates it
  • Anti-entropy: background process syncs replicas
  • Sloppy quorum: write to available nodes even if not the home replicas (hinted handoff)

Failover & Replication Lag Problems

  • Failover pitfalls: async replication means new leader may be behind → lost writes. Set old leader as follower to recapture
  • Read-your-own-writes: user writes, then reads from lagging follower, sees old data. Fix: route reads of user's own content to leader
  • Monotonic reads: user reads from replica A (recent), then replica B (stale) → time goes backwards. Fix: sticky sessions to same replica
  • Consistent prefix reads: causally related writes must be read in order

Likely interview questions

  • Explain the difference between synchronous and asynchronous replication. What are the trade-offs?
  • What is replication lag and what problems does it cause? How would you mitigate it?
  • How does Cassandra handle writes with no single leader? What is a quorum?
  • How would you handle a leader failover to avoid split-brain or lost writes?
CH 06

Partitioning (Sharding)Splitting data across multiple nodes to scale beyond a single machine

Key-Range Partitioning range queries

  • Partition by key range: A–F on shard 1, G–M on shard 2, etc.
  • Pro: efficient range queries (scan one partition)
  • Con: Hot spots — if everyone's writing timestamps, today's partition gets all the traffic
  • Used by: HBase, BigTable, MongoDB (range-based)

Hash Partitioning even distribution

  • Hash the key, assign to partition based on hash range. Even load distribution
  • Con: range queries now span all partitions
  • Cassandra: compound key — hash on first part, range on rest (best of both)
  • Celebrity hot-spot problem: famous user has millions of followers. Solution: add random prefix to key, then aggregate reads

Rebalancing Partitions elasticity

  • Fixed number of partitions: create 1000 partitions upfront, move whole partitions between nodes (Elasticsearch, Riak)
  • Dynamic partitioning: split when partition exceeds threshold, merge when too small (HBase, RethinkDB)
  • Consistent hashing: arrange nodes on a ring. Add/remove node → only adjacent partitions move. Minimises data movement
  • Virtual nodes (vnodes): each physical node owns many small virtual partitions on the ring → smoother rebalancing (Cassandra)

Secondary Indexes & Partitioning scatter-gather problem

  • Local (document-based) secondary index: each partition indexes its own data. Reads must query all partitions and merge (scatter-gather). Used by MongoDB, Cassandra, Elasticsearch
  • Global (term-based) secondary index: index itself is partitioned separately. Writes update multiple partitions; reads hit one partition. Used by DynamoDB GSIs

Likely interview questions

  • How would you shard a user table across 10 database nodes? What's the trade-off between range and hash sharding?
  • You notice one shard is receiving 80% of writes. What's causing this and how do you fix it?
  • What is consistent hashing and why do distributed systems use it?
  • How do secondary indexes work across partitions?

TransactionsACID, isolation levels, and the concurrency problems they prevent — critical for fintech interviews

ACID — what each property actually means interview essential

PropertyWhat it guaranteesReal example
AtomicityAll operations in a transaction succeed, or none do. No partial writes.Debit €100 AND credit €100 — either both happen or neither does
ConsistencyTransaction brings DB from one valid state to another (application-defined invariants)Account balance must never go negative after any transaction

Isolation Levels — from weakest to strongest core concept

LevelPreventsAllows (problem)Used by default in

Concurrency Problems — be able to explain these anomalies

  • Dirty read: reading another transaction's uncommitted data
  • Non-repeatable read: same row returns different values within one transaction (another tx committed in between)
  • Phantom read: same query returns different rows (another tx inserted/deleted matching rows)
  • Write skew: two transactions read overlapping data, each makes a decision, both write — combined result violates an invariant. Classic example: two doctors both go off-call when at least one must remain
  • Lost update: read-modify-write cycle where one transaction's write overwrites another's

Implementing Serializable Isolation implementation

  • Actual Serial Execution: one thread, one transaction at a time. Fast if transactions are short and data fits in memory. Used by VoltDB, Redis (single-threaded)
  • Two-Phase Locking (2PL): readers block writers and vice versa. Pessimistic. Deadlock risk. Used by MySQL, older PostgreSQL
  • Serializable Snapshot Isolation (SSI): optimistic — detect conflicts at commit time and abort. Used by PostgreSQL ≥9.1, CockroachDB. Better throughput than 2PL under low contention

Distributed Transactions — 2PC and its problems distributed

When a transaction spans multiple nodes or services, you need distributed coordination.

  • Two-Phase Commit (2PC): Coordinator sends "prepare" to all participants → all vote yes/no → coordinator sends "commit" or "abort"
  • Problem: coordinator failure after "prepare" but before "commit" leaves participants in doubt — they hold locks, blocking progress indefinitely
  • XA transactions: standard 2PC across heterogeneous systems. Widely supported but heavyweight
  • Alternative — Saga pattern: break transaction into sequence of local transactions with compensating actions for rollback. Used widely in microservices (Stripe, Airbnb)
# Saga example: booking a flight + hotel Step 1: Book flight → success Step 2: Book hotel → FAILS Compensate: Cancel flight ← compensating transaction

Likely interview questions

  • Explain ACID. What does isolation actually mean in practice?
  • What is write skew? Give a real example. How do you prevent it?
  • What is the difference between Snapshot Isolation and Serializable Isolation?
  • Why is 2PC problematic in distributed systems? What are the alternatives?
  • How would you handle a payment that needs to debit one service and credit another (across microservices)?
CH 08

The Trouble with Distributed SystemsPartial failures, unreliable networks, and clocks — the reality of distributed systems

Partial Failures the hard part

In a single computer, something either works or fails — you can tell which. In a distributed system, a node might be slow, a network packet might be lost, or a node might appear dead from one side and alive from another. You cannot know whether a remote operation succeeded.

  • Timeouts: the only way to detect dead nodes — but a timed-out request might have actually succeeded
  • Idempotency: design operations that are safe to retry. Use idempotency keys (Stripe does this)

Unreliable Clocks dangerous assumption

  • Wall clock: can jump backwards (NTP sync). Never use for ordering events across nodes
  • Monotonic clock: only for measuring duration on a single machine — no meaning across nodes
  • Last-write-wins with wall clocks silently drops data when clocks disagree — the default in Cassandra
  • Google TrueTime: returns [earliest, latest] with bounded uncertainty. Spanner waits out the uncertainty before committing
  • Logical clocks (Lamport): counters that capture causal ordering without wall time. Safe for ordering

Process Pauses — why distributed locks are hard subtle

A process can pause at any point: GC stop-the-world, VM migration, paging. This means:

  • A process holding a lease can pause, let the lease expire, resume — and still think it holds the lock
  • Fencing tokens: each lock grant comes with a monotonically increasing token. Storage layer rejects writes with an older token. Safe even if the lock holder pauses
# Without fencing — dangerous Client 1: acquires lock, pauses for 30s (GC) Client 2: lock expires, acquires lock, writes data Client 1: resumes, thinks it still has lock, overwrites Client 2's data ← CORRUPTION # With fencing tokens — safe Client 1: gets lock with token=33, pauses Client 2: gets lock with token=34, writes (storage records min_token=34) Client 1: resumes, tries to write with token=33 → storage REJECTS (33 < 34) ✓

Likely interview questions

  • Why can't you rely on wall clock timestamps to order events across distributed nodes?
  • What is a fencing token and why do you need it for distributed locks?
  • A network request times out. How do you know if the operation actually happened? How do you handle this?
CH 09

Consistency & ConsensusCAP theorem, linearizability, and how distributed agreement actually works

CAP Theorem often misunderstood

In the presence of a network Partition, you must choose between Consistency (linearizability) and Availability. CP or AP — not both.

⚠️ Interview trap: CAP is widely misunderstood. "Consistency" in CAP means linearizability, not eventual consistency or ACID consistency. And network partitions are rare — most of the time systems operate in the CA region. The more useful framing is the PACELC theorem: even without partitions, there's a latency vs consistency trade-off.
SystemChoice under partition
Zookeeper, etcd, HBaseCP — rejects writes if can't achieve quorum. Consistent but may be unavailable
Cassandra, DynamoDB, CouchDBAP — always accepts writes. Eventually consistent
PostgreSQL (single node)Not distributed — N/A. But if using replication, depends on config

Linearizability strongest guarantee

The system appears to have a single, atomic register. Every read returns the most recently written value. Operations look instantaneous.

  • Required for: leader election, distributed locks, uniqueness constraints, atomic compare-and-swap
  • Cost: requires cross-node coordination on every operation → high latency
  • Multi-leader and leaderless systems are generally not linearizable
  • Single-leader with sync replication can be linearizable if you always read from leader

Eventual Consistency availability-first

If you stop writing, all replicas will eventually converge to the same value. No timing guarantee on when.

  • Weaker but scales better — no need for coordination on reads/writes
  • Monotonic read consistency: you'll never see older data after seeing newer data
  • Read-your-writes: after writing, you see your own write
  • These are weaker guarantees you can layer on top of eventual consistency

Consensus Algorithms — Raft & Paxos how it really works

Consensus = getting multiple nodes to agree on a single value, despite faults. Required for: leader election, atomic broadcast, distributed lock services.

  • Paxos: the original consensus algorithm. Notoriously hard to understand. Lamport's 1998 paper
  • Raft: designed to be understandable. Divides problem into leader election + log replication + safety. Used by etcd, Consul, CockroachDB, TiKV
  • Raft leader election: node with highest term and most up-to-date log wins election by getting votes from majority
  • Raft log replication: leader appends entries to log, sends to followers, commits once majority ack. Committed = durable even if leader fails
  • Multi-Paxos / Viewstamped Replication: variations used in Spanner, Chubby
Raft Leader Election Node 1 (follower): timeout → becomes candidate → sends RequestVote Node 2: votes yes (has older log) Node 3: votes yes → Node 1 wins with 2/3 majority → becomes leader Raft Log Replication Leader: appends entry [index=5, term=3, cmd="set x=1"] to its log Leader: sends AppendEntries to followers Followers: ack receipt Leader: once 2/3 ack → marks entry committed → applies to state machine

ZooKeeper & etcd — coordination services building block

These implement consensus and expose primitives that other distributed systems build on:

  • Leader election: a service creates an ephemeral node; first one to do so is the leader. Others watch for deletion
  • Distributed locks: sequential ephemeral znodes. Lowest-numbered node holds the lock
  • Service discovery: services register themselves; clients watch for changes
  • Config store: store cluster config; all nodes watch for updates
  • Kubernetes uses etcd for all cluster state. Kafka used ZooKeeper (now moving to KRaft — Kafka's own Raft)

Likely interview questions

  • Explain CAP theorem. Where does Cassandra sit vs Zookeeper?
  • What is linearizability? When do you need it vs when can you accept eventual consistency?
  • How does Raft achieve consensus? What happens when the leader fails?
  • How would you implement distributed leader election without Zookeeper?
CH 10

Batch ProcessingMapReduce, Spark, and the Unix philosophy applied to data at scale

MapReduce — the conceptual foundation foundational pattern

A programming model for processing large datasets in parallel across a cluster. Two phases: Map (transform each record) and Reduce (aggregate by key).

# Word count in MapReduce Map: "hello world hello" → [("hello",1), ("world",1), ("hello",1)] Shuffle: → [("hello",[1,1]), ("world",[1])] Reduce: → [("hello",2), ("world",1)] # Key insight: map outputs are sorted by key, sent to reducers # The shuffle (sort-merge join across network) is the expensive part
  • HDFS stores data in chunks across nodes. MapReduce moves computation to the data, not data to computation
  • Fault tolerant: if a task fails, re-run it. Output is deterministic (pure functions)
  • Write output to HDFS → next job reads it. Chaining jobs = data pipelines

Beyond MapReduce — Spark in-memory batch

  • Spark keeps intermediate results in memory (RDDs) instead of writing to HDFS between stages → 10–100x faster than MapReduce for iterative algorithms
  • DAG of transformations: lazy evaluation — operations build a plan, executed when action triggered
  • Spark SQL / DataFrames: declarative API on top of Spark engine
  • Fault tolerance via lineage: can recompute lost partitions by replaying transformations
  • Used for: ETL, ML training, large-scale analytics, nightly batch jobs

Join Strategies at Scale practical

  • Sort-merge join: both sides sorted by join key, merged in one pass. Used in MapReduce reduce phase
  • Broadcast join (map-side): small table broadcast to all workers, large table joined locally. No shuffle. Used when one table fits in memory
  • Hash join: build hash table from smaller side, probe with larger side. Fastest when data fits in memory
  • Skew handling: hot keys make one reducer a bottleneck. Fix: salt the key, process hot keys separately

Lambda & Kappa Architecture architectural patterns

  • Lambda: run batch layer (accurate, slow) + speed layer (approximate, fast) in parallel. Query layer merges results. Complex to maintain two codebases
  • Kappa: only a stream processing layer. Replay historical data through streaming system to "reprocess". Simpler — one codebase. Used by LinkedIn, Kafka-native pipelines

Likely interview questions

  • Explain MapReduce. How would you count unique users per day across 1TB of logs?
  • Why is Spark faster than MapReduce? What are the trade-offs?
  • What is a broadcast join and when would you use it?
  • Compare Lambda vs Kappa architecture for a real-time + historical reporting system.
CH 11

Stream ProcessingKafka, windowing, exactly-once semantics — processing data as it arrives

Apache Kafka — the de facto event streaming platform essential knowledge

Producer → Topic: payments (partitioned across brokers) ├── Partition 0: [offset 0][offset 1][offset 2]... ├── Partition 1: [offset 0][offset 1]... └── Partition 2: [offset 0][offset 1]... ↓ Consumer Group A (fraud detection) — each partition consumed by one consumer Consumer Group B (analytics) — independent offset, replays from start
  • Log structure: messages are appended to an immutable, ordered log. Consumers track their own offset — Kafka doesn't track per-consumer state (unlike RabbitMQ)
  • Retention: messages retained for configurable period (days/weeks) — consumers can re-read or replay
  • Partitions = parallelism: each partition consumed by one consumer in a group. More partitions → more parallelism
  • Ordering guarantee: ordering guaranteed within a partition. Use consistent hashing on key to ensure related events land on same partition
  • Compaction: log-compacted topic keeps only the latest value per key — useful as a changelog or source of truth
  • Replication: each partition has leader + followers. Producer writes to leader, followers replicate. ISR (in-sync replicas) must ack for "all" durability setting

Delivery Semantics — the most important trade-off in streaming interview essential

SemanticWhat it meansRiskHow achieved
At-most-onceMessage delivered 0 or 1 timesMessage lossFire-and-forget, no retry
At-least-onceMessage delivered 1 or more timesDuplicatesRetry on failure; consumer must be idempotent
Exactly-onceMessage delivered exactly onceComplexityIdempotent producer + transactional API (Kafka) or Flink checkpoints
💡 Real talk: "exactly-once" is technically possible in Kafka (with transactions) but it's expensive and complex. Most systems accept at-least-once + idempotent consumers as the pragmatic choice. Stripe uses idempotency keys for exactly this reason.

Windowing — time in stream processing key concept

  • Tumbling window: fixed, non-overlapping. "Sales total every 5 minutes"
  • Sliding window: overlapping. "Average over last 5 minutes, updated every 1 minute"
  • Session window: grouped by user activity gap. "All events within 30min of each other"
  • Event time vs processing time: events arrive late (network delay, mobile offline). Use watermarks to track how far behind processing is, and decide when to close a window

Stream-Table Duality & Joins advanced

  • A table is a stream of changes: each row update is an event. A stream is a table at each point in time. This unifies batch and stream processing
  • Stream-stream join: join two event streams within a time window (e.g. join click stream with purchase stream within 1hr)
  • Stream-table join: enrich events with a slowly-changing table (e.g. enrich payment events with current user tier). Table acts as a lookup
  • Table-table join: materialise join of two tables, update incrementally. Complex — needs careful state management

Apache Flink — stateful stream processing production streaming

  • Stateful operator: each event updates in-memory state. Flink checkpoints state to durable storage (HDFS/S3) periodically for fault tolerance
  • Exactly-once: Chandy-Lamport algorithm — barriers flow through the DAG, state snapshot taken at barrier. On failure, replay from last checkpoint
  • vs Spark Streaming: Flink is true streaming (event-at-a-time). Spark Structured Streaming is micro-batching. Flink has lower latency; Spark has better SQL tooling
  • vs Kafka Streams: Kafka Streams is a library (runs in your app), no separate cluster. Simpler ops, limited to Kafka data sources. Flink has richer operator set and better windowing

Change Data Capture (CDC) — real-world data integration practical pattern

CDC captures every insert/update/delete from a database's write-ahead log and streams them as events. Used to sync data between systems without polling.

# Example: Postgres → Kafka via Debezium (CDC tool) Postgres WAL: INSERT INTO orders (id, amount) VALUES (123, 99.99) ↓ Debezium reads WAL Kafka topic: postgres.public.orders { "op": "c", "after": {"id": 123, "amount": 99.99}, "ts": 1718000000 } ↓ consumers Search index (Elasticsearch) Analytics DB (BigQuery) Cache (Redis)

Use CDC instead of dual-writes (writing to DB + Kafka in same request) because dual-writes create inconsistency if one fails. CDC guarantees the Kafka event is emitted if and only if the DB write committed.

  • How does Kafka guarantee ordering? What happens with multiple partitions?
  • Explain at-least-once vs exactly-once delivery. Which would you choose for a payments system and why?
  • What is a watermark in stream processing? Why do you need it?
  • How would you build a real-time fraud detection system using Kafka and Flink?
  • What is CDC? Why is it better than dual-writes for syncing between a database and a message queue?

CH 12

The Future of Data SystemsUnbundling databases, dataflow systems, and correctness at scale

Unbundling the Database architectural insight

Kleppmann's thesis: the monolithic database bundles storage, indexing, replication, query execution. Modern systems unbundle these into specialised components connected by streams.

  • Postgres for transactions → CDC → Kafka → Elasticsearch for search, Redis for cache, BigQuery for analytics
  • Each component does one thing well
  • The event log (Kafka) becomes the system of record; derived data stores are just views

Derived Data & Idempotence correctness

  • Any derived data (cache, search index, ML feature store) should be reproducible by replaying the source event log
  • Operations should be idempotent — safe to re-run on replay
  • Immutable event log as source of truth lets you rebuild any downstream system from scratch — the ultimate debuggability

End-to-end Correctness senior engineering mindset

Low-level guarantees (Kafka exactly-once, database ACID) are not enough on their own. You must ensure correctness end-to-end across the full application.

  • A payment idempotency key must be checked at the application layer, not just the database layer
  • Distributed systems will have partial failures — design for it explicitly with operation IDs, idempotency, and reconciliation
  • Auditing: append-only ledger of all events makes it possible to reconstruct state and detect discrepancies — the approach used by all financial systems

Likely interview questions

  • If you were designing a data platform from scratch for a fintech at scale, how would you structure the data flow?
  • Why is an immutable event log a powerful architectural primitive? What can you build on top of it?
  • How do you ensure a distributed payment system doesn't double-charge users even under partial failures?
