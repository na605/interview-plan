What it is

A hashmap stores key→value pairs with O(1) average lookup and insert. In Java it's HashMap<K,V>. In Python it's a dict. The core trick: instead of searching for something, pre-store it and check existence in O(1).

The pattern signal

If the problem asks "does X exist", "have I seen this before", "how many times does Y appear", or "find a pair that satisfies condition Z" — it's almost always a hashmap problem.

Two Sum — the canonical example

Problem: given an array, find two indices whose values sum to target.

Brute force: O(n²) — check every pair. Too slow.

Hashmap approach: as you scan, store complement → index. For each number, check if its complement already exists in the map.

// Java — Two Sum O(n)
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[]{map.get(complement), i};
        }
        map.put(nums[i], i); // store AFTER check to avoid using same element twice
    }
    return new int[]{};
}

Prefix Sum + Hashmap — Subarray Sum = K

Problem: count subarrays with sum equal to k.

Key insight: if prefix[j] - prefix[i] = k, then there's a subarray from i+1 to j with sum k. So we check: has (currentSum - k) appeared before?

// Java — Subarray Sum Equals K O(n)
public int subarraySum(int[] nums, int k) {
    Map<Integer, Integer> map = new HashMap<>();
    map.put(0, 1); // empty prefix has sum 0
    int count = 0, sum = 0;
    for (int num : nums) {
        sum += num;
        count += map.getOrDefault(sum - k, 0);
        map.put(sum, map.getOrDefault(sum, 0) + 1);
    }
    return count;
}

Edge cases to always check

Empty array → return 0 or [] immediately
Single element → does it equal target on its own?
Negative numbers → prefix sum still works, but k=0 may find unexpected matches
Duplicates → don't overwrite the map entry on second occurrence (Two Sum: store first index only)

When to use it

Signal words: "contiguous subarray", "substring", "window of size k", "longest/shortest segment that satisfies...". If a brute force would check every subarray in O(n²), sliding window gives you O(n).

Two types

  • Fixed window: window size is given (e.g. "maximum average subarray of length k"). Slide by adding the new element and removing the old one.
  • Variable window: window size changes based on a condition (e.g. "longest substring without repeating chars"). Expand right pointer, shrink left when condition breaks.

Variable window template

// Generic variable sliding window — Java
int left = 0, result = 0;
Map<Character, Integer> window = new HashMap<>();

for (int right = 0; right < s.length(); right++) {
    char c = s.charAt(right);
    window.put(c, window.getOrDefault(c, 0) + 1);

    // shrink from left while condition is violated
    while (window.get(c) > 1) {
        char leftChar = s.charAt(left);
        window.put(leftChar, window.get(leftChar) - 1);
        left++;
    }

    result = Math.max(result, right - left + 1);
}
return result;

Product of Array Except Self — no division trick

Can't use division (what if there's a zero?). Instead, compute prefix products left-to-right, then suffix products right-to-left, multiply them together.

// O(n) time, O(1) extra space (output array doesn't count)
public int[] productExceptSelf(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];

    // left pass: result[i] = product of everything to the left
    result[0] = 1;
    for (int i = 1; i < n; i++)
        result[i] = result[i-1] * nums[i-1];

    // right pass: multiply in suffix product on the fly
    int suffix = 1;
    for (int i = n-1; i >= 0; i--) {
        result[i] *= suffix;
        suffix *= nums[i];
    }
    return result;
}

Edge cases

All zeros in array → every product except self is 0
One zero → only the position of the zero gets a non-zero result
Negative numbers → sign flips, algorithm still works

BFS (Breadth-First Search)

Explores level by level using a queue. Use when you need the shortest path, minimum steps, or need to process nodes level by level.

Signal words: "minimum distance", "shortest path", "nearest", "level order", "fewest moves".

// BFS template — Java
Queue<int[]> queue = new LinkedList<>();
boolean[][] visited = new boolean[rows][cols];
queue.offer(new int[]{startRow, startCol});
visited[startRow][startCol] = true;
int steps = 0;

int[][] dirs = {{0,1},{0,-1},{1,0},{-1,0}};

while (!queue.isEmpty()) {
    int size = queue.size(); // process one level at a time
    for (int i = 0; i < size; i++) {
        int[] cell = queue.poll();
        if (cell[0] == endRow && cell[1] == endCol) return steps;
        for (int[] d : dirs) {
            int r = cell[0] + d[0], c = cell[1] + d[1];
            if (r >= 0 && r < rows && c >= 0 && c < cols && !visited[r][c]) {
                visited[r][c] = true;
                queue.offer(new int[]{r, c});
            }
        }
    }
    steps++;
}

DFS (Depth-First Search)

Explores as deep as possible before backtracking — using recursion (or an explicit stack). Use when you need to explore all paths, count connected components, or check if a path exists.

Signal words: "number of islands", "connected components", "all paths", "exists a path".

// DFS on a grid — Java (recursive)
private void dfs(char[][] grid, int r, int c) {
    if (r < 0 || r >= grid.length || c < 0 || c >= grid[0].length
        || grid[r][c] != '1') return;

    grid[r][c] = '0'; // mark visited in-place
    dfs(grid, r+1, c);
    dfs(grid, r-1, c);
    dfs(grid, r, c+1);
    dfs(grid, r, c-1);
}

Tree DFS — recursion pattern

// Max depth — classic base case pattern
public int maxDepth(TreeNode node) {
    if (node == null) return 0;
    return 1 + Math.max(maxDepth(node.left), maxDepth(node.right));
}

Multi-source BFS — Rotting Oranges pattern

Seed the queue with all starting nodes at once before the BFS loop. This simulates simultaneous spread from multiple sources.

// Seed queue with ALL rotten oranges before BFS starts
for (int r = 0; r < rows; r++)
    for (int c = 0; c < cols; c++)
        if (grid[r][c] == 2) queue.offer(new int[]{r, c});
// Then run standard BFS — time = number of levels

When to use it

Any problem involving ranges, meetings, time slots, or overlapping segments. Amazon loves logistics-themed variants: "delivery time windows", "meeting room conflicts", "server maintenance windows".

Algorithm

  • Sort by start time
  • Iterate — if current interval overlaps with last merged (current.start ≤ last.end), extend last.end
  • Otherwise, add current to result
public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    List<int[]> result = new ArrayList<>();

    for (int[] interval : intervals) {
        if (result.isEmpty() || result.get(result.size()-1)[1] < interval[0]) {
            result.add(interval); // no overlap
        } else {
            result.get(result.size()-1)[1] = Math.max(
                result.get(result.size()-1)[1], interval[1]); // merge
        }
    }
    return result.toArray(new int[0][]);
}
Single interval → return it as-is
All intervals overlap → result has one entry spanning min start to max end
Adjacent intervals [1,2] and [2,3] → these DO overlap (end == next start)

The format

You get a real multi-file project — a React app, Express server, or similar. It already has working code. You're given 2 tasks to complete inside it, like:

  • Fix a bug in an existing function (null check, off-by-one, wrong comparison)
  • Add a new feature or endpoint (e.g. filter orders by status, add input validation)

The IDE has a file explorer, editor, terminal, run/preview pane, and an AI assistant tab. You run tests by clicking "Run Tests" in the IDE.

Key differences from the algorithmic question

  • No blank editor — you start with existing code to read and understand
  • No algorithm to invent — you need to navigate and modify existing logic
  • AI assistant is available and your prompts are logged by the evaluator
  • Tests are automated — pass/fail visible in the terminal

How you're scored

Tests pass/fail (visible + hidden). But also: the evaluator reads your AI assistant conversation. Specific, targeted prompts show strong engineering instinct. Vague prompts like "explain this code" look weak.

What the assistant can do

  • Explain what a specific function or file does
  • Help you navigate: "which file handles order routing?"
  • Give syntax help: "what's the Node.js syntax for reading a query param?"
  • Debug: "I'm getting TypeError: Cannot read property 'id' of undefined — what's likely wrong?"
  • Suggest code snippets (you paste them manually)

What it can't do

  • Generate a complete working solution end-to-end
  • Run code or read test output for you
  • Make decisions about what to implement

Good vs bad prompts — examples

❌ Weak prompt
"Explain this code"
Why it's weak
Vague. Shows no prior reading. The evaluator sees this and flags it as low signal.
✅ Strong prompt
"In routes/orders.js, the getOrderById function — what does it return if the ID doesn't exist in the database?"
Why it's strong
File-specific, function-specific, asking about a real edge case. Shows you've already read the code and have a specific question.
✅ Strong prompt
"I need to add a filter by status parameter to GET /orders. In this Express app, how do I read a query param and pass it to the service layer?"
Why it's strong
Shows you know the architecture (route → service layer), you've identified the right file, and you just need syntax help — not design help.
✅ Strong prompt
"Running tests gives me AssertionError: expected 404 but got 200. The route exists but returns 200 even for missing IDs. Where should the null check go?"
Why it's strong
You've run the tests, read the error, formed a hypothesis, and are asking a targeted question. That's how good engineers use tools.

Strategy for navigating an unfamiliar codebase fast

  • Start with the entry point — index.js, app.js, server.js, or main.java
  • Follow the route — find where the relevant endpoint is registered, then the handler, then the service
  • Read the test file first — it tells you exactly what behaviour is expected
  • Don't read everything — you only need the files relevant to your task

// Bug: crashes if user is null

// Fix

// Bug: returns one too many items

// Fix

// Bug: returns 200 when item not found

// Fix

// Bug: continues after sending error response

// Fix: add return
if (!user) return res.status(401).send('Unauthorized');

Study material

You don't need to memorise all 16. For the SDE work simulation, these 5 are tested most heavily. Read each one, understand what Amazon means by it, and see the answer signals.

1. Customer Obsession
Start with the customer and work backwards. When there's a conflict between internal convenience and user experience, the user wins. The "customer" in an SDE context can be an internal team, not just an end user.
Answer signal: when two options look similar, pick the one that prioritises user-facing impact over internal process or engineering preference.
2. Ownership
Act on behalf of the entire company, not just your immediate team. Don't say "that's not my problem." If you see a bug outside your scope, you flag it or fix it. If something breaks, you take responsibility even if you didn't cause it.
Answer signal: pick the answer where YOU take action — not the one where you wait for someone else, escalate without trying first, or blame another team.
3. Bias for Action
Speed matters. Many decisions are reversible. Make a reasonable decision with imperfect information rather than waiting for certainty. Calculated risk is valued over endless analysis.
Answer signal: when the choice is "wait for complete info / more meetings" vs "take a reasonable action now with what you know" — pick the action. Exception: genuinely irreversible decisions with high risk.
4. Deliver Results
Focus on key inputs and deliver on time. Don't let blockers go unaddressed. If a deadline is at risk, surface it early — going quiet is the worst option.
Answer signal: raise blockers early, flag risks proactively, don't silently miss a deadline and hope no one notices. Communicate, don't hide.
5. Dive Deep
Stay connected to the details. Check the data before accepting a surface-level explanation. If something seems wrong, dig into why before deciding what to do.
Answer signal: when there's an option to check the data / investigate the root cause vs accept the first explanation — check the data.

The format

You get a simulated inbox with emails, instant messages, and data attachments from virtual teammates and managers. Each scenario presents 3–5 response options. You pick one (or rank them). There's no single "correct" answer displayed — you're scored against LP alignment.

Question types you'll see

  • Prioritisation: Two urgent things arrive at once. Which do you handle first?
  • Conflict: A teammate has done something wrong. How do you respond?
  • Ambiguity: Requirements are unclear and you can't reach anyone. What do you do?
  • Trade-off: Ship on time with a known bug, or delay for quality?
  • Scope: You discover a problem outside your team's ownership. What do you do?

The answer-picking framework

When two options look similar, ask:

  • Which one has me taking action rather than waiting or delegating?
  • Which one prioritises the customer over internal comfort?
  • Which one raises the problem early rather than hiding it?
  • Which one is backed by data or investigation rather than gut feel?

What it is

~38 paired statements. For each pair, you choose which is "Most Like Me" vs "More Like Me" (or a Likert scale variant). The content covers decision-making, collaboration, risk appetite, and work preferences.

The critical trap: inconsistency

Amazon runs consistency checks across responses. If you say "I always follow established processes" early on, then "I'd rather act fast than wait for process" later, both marked "Most Like Me" — that's flagged. Don't try to answer each question in isolation as "what Amazon wants". Answer as a consistent person.

The profile Amazon wants to see

  • Takes initiative over waiting for direction
  • Raises concerns and blockers early rather than staying quiet
  • Values data and evidence over pure intuition
  • Cares about long-term quality, not just short-term output
  • Owns mistakes rather than attributing them to circumstances
  • Adapts when priorities shift — doesn't rigidly follow a plan when context changes
  • Learns from failure rather than being discouraged by it

Sample questions with reasoning

Question type
"I prefer to gather all available information before making a decision" vs "I'm comfortable making decisions before having all the information"

Lean toward the second — Bias for Action. "Most Like Me" unless the risk is genuinely catastrophic. Amazon values reasonable action with imperfect info over analysis paralysis.

Scenario 1 · Ownership + Bias for Action
A teammate merged a breaking change and went on holiday. The incident is live. You didn't cause it and don't own their service. What do you do?
Model answer
You take ownership. Investigate the breaking change, understand the scope, and either roll back or patch it. Notify your manager proactively. Don't wait for the teammate to return or for someone else to take it. LP: Ownership — act on behalf of the entire company, not just your remit. Bias for Action — don't wait; the incident is live.
Scenario 2 · Deliver Results + Earn Trust
Your manager wants a feature shipped by Friday. You estimate it needs 2 more days of testing. Missing the deadline would affect a partner team. What do you do?
Model answer
Surface it immediately — don't go quiet and hope to close the gap. Communicate the risk, your current estimate, and what you've done to try to close it. Propose options: ship with reduced scope, get help, or delay with clear reason. LP: Deliver Results — flag blockers early. Earn Trust — transparency builds it; silence destroys it. Never ship something you know is broken without disclosure.
Scenario 3 · Ownership beyond scope
While reviewing a teammate's PR you notice a performance issue that will affect customers but is outside your team's ownership. The teammate hasn't noticed. What do you do?
Model answer
Flag it in the PR review with a clear explanation of the impact. If the teammate is unresponsive, escalate to the owning team directly. Don't merge it silently because "it's not your problem." LP: Ownership — think beyond your immediate scope. Customer Obsession — the performance issue affects customers; that's reason enough to act.
Scenario 4 · Earn Trust + Customer Obsession
A stakeholder emails asking why a feature is two weeks late. The delay was caused by a dependency team, not yours. How do you respond?
Model answer
Don't deflect or blame the dependency team in your response. Acknowledge the delay, take responsibility for the overall outcome, explain the root cause factually, and give a clear updated timeline. LP: Earn Trust — stakeholders trust people who own outcomes, not people who point fingers. Deliver Results — you own the delivery, not just your slice of it.
Scenario 5 · Customer Obsession vs delivery pressure
You're asked to cut scope on a feature to hit a deadline. The cut would meaningfully hurt the end user experience. Leadership is pushing to ship. What do you do?
Model answer
Raise the user experience concern explicitly with data if you have it ("this removes X from N users who rely on it"). Propose an alternative: ship the core feature now with a clear follow-up plan for the cut scope. If leadership still decides to ship, commit and execute — but your job is to make sure they're making the decision with full information. LP: Have Backbone — disagree and say so. Customer Obsession — advocate for the user. Then Disagree and Commit once the decision is made.
Scenario 6 · Bias for Action under ambiguity
You're mid-task and realise the requirements are ambiguous in a way that could mean two completely different implementations. Your manager is in back-to-backs for 3 hours and the deadline is today. What do you do?
Model answer
Don't freeze. Make the most reasonable interpretation based on context, document your assumption explicitly, and implement it. Send your manager a concise Slack/email: "Implemented X based on assumption Y — please flag if that's wrong." This shows Bias for Action (you moved forward) and Earn Trust (you were transparent). The worst answer is to do nothing and wait 3 hours.

Browser rules (enforced automatically)

  • Copy-paste is flagged — even your own code. Type it. Amazon expects original code written in real time
  • Print Screen ends the test immediately — don't press it

Setup checklist before you start

  • Latest Chrome (version 94+)
  • Disable all browser extensions
  • Enable third-party cookies
  • Stable internet (4G minimum) — check speed beforehand
  • Quiet uninterrupted room for the full duration
  • Water, no phone in sight

During the test — time management

  • Read the whole problem before writing a single line
  • Write a brute force solution comment first — it clarifies your thinking
  • If stuck at 20 mins: write the brute force that passes visible test cases, then optimise. Partial credit > blank.
  • Test edge cases before submitting: empty input, single element, negatives, large n
  • If you finish Q1 early, move to Q2 — don't micro-optimise

Amazon's hidden test cases heavily target edge conditions. Before submitting, run through this mentally:

Array / string problems

Empty array/string → return 0, [], or "" immediately?
Single element → does it satisfy the condition on its own?
All same elements → does your algorithm handle duplicates?
Negative numbers → prefix sum, product calculations still work?
Integer overflow → use long in Java if multiplying large numbers

Graph / grid problems

Disconnected graph → does your BFS/DFS handle multiple components?
1×1 grid → single cell edge case
All cells blocked / all open → boundary conditions
Already visited tracking → are you marking visited before or after adding to queue? (Add BEFORE to avoid re-adding)

Intervals

Single interval → return as-is
Adjacent intervals [1,2] [2,3] → these overlap (use ≤ not <)
All overlapping → result is one merged interval

1. Where did time go?

Reading the problem? Planning the approach? Debugging? If you spent more than 10 mins reading — practice skimming. If you spent more than 15 mins debugging — write brute force first next time.

2. Did I check edge cases before submitting?

If no — that's one habit to add on test day. 2 mins of edge case checking can convert partial credit to full credit.

3. Did I write brute force first or go straight for optimal?

Going straight for optimal when rusty causes freezing. On test day: write the brute force in comments first (even if you don't code it), then build the optimal. The act of writing it clarifies your thinking.
