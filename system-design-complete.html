CH 01

Reliability, Scalability & MaintainabilityThe three properties every system must be designed around

Reliability fault-tolerant

  • Hardware faults: RAID, redundant power, multi-AZ replicas
  • Software faults: chaos testing, careful error handling, isolation
  • Human error: sandboxes, gradual rollouts, fast rollback, monitoring

Scalability load & performance

  • Describe load with precise parameters (e.g. Twitter: 12 writes/s, 345K reads/s)
  • Throughput (batch) vs latency (online: p99, not mean)
  • p99 matters — tail latency amplifies across microservices

Maintainability operability · simplicity · evolvability

  • Operability: good monitoring, runbooks, config management
  • Simplicity: reduce accidental complexity
  • Evolvability: rolling deploys, feature flags, no big-bang rewrites
Interview Questions & Model Answers — click to expand
▶How would you design a system to handle 10x traffic growth with no downtime?
Model Answer
  • Make the web tier stateless first. Session state must live in Redis, not in process memory. Without this, you can't freely add/remove web server instances — requests must always hit the same server, which kills horizontal scaling.
  • Horizontal scale over vertical. Vertical scaling (bigger machine) has a ceiling and is a single point of failure. Horizontal scaling behind a load balancer is elastic — add nodes under load, remove them after.
  • Read replicas before sharding. If read throughput is the bottleneck, replicas are the lowest-friction solution. Only shard when write throughput is the constraint — sharding adds significant operational complexity.
  • Cache aggressively. Redis cache-aside in front of the DB can absorb 80–90% of reads if data has temporal locality. Profile before caching — not everything benefits equally.
  • CDN for static assets. Images, JS, CSS should never hit your origin at 10x load. A CDN absorbs this at the edge for near-zero marginal cost per request.
  • Async for non-critical work. Email, notifications, report generation — queue via Kafka or SQS and process with workers you scale independently of the request path.
  • Measure before optimising. Load test to find the actual bottleneck. It's almost never where you expect it to be.
Real example — Stripe handling Black Friday: - Pre-scale web tier based on YoY historical patterns (known spike) - Rate-limit non-critical endpoints (analytics, reporting APIs) - Shed non-essential async work during peak (defer batch jobs) - Keep payments critical path isolated from all other services - Chaos-test failover paths weeks before the event
▶Why do engineers care about p99 latency, not the mean?
Model Answer
  • The mean hides tail misery. If 99% of requests take 10ms and 1% take 10 seconds, the mean is ~110ms — which sounds fine. But at 10M requests/day, that's 100,000 terrible user experiences.
  • p99 = the worst case for almost everyone. 99th percentile means 99% of requests are faster than this value. It's the number that drives SLA commitments.
  • Tail latency amplifies in microservices. If a single request fans out to 10 downstream services and waits for all of them, the request latency is the max across all 10, not the mean. If each service has p99=20ms, the combined p99 approaches 200ms. Kleppmann calls this "tail latency amplification."
Service A: p50=5ms, p99=100ms Service B: p50=4ms, p99=80ms Combined (both called sequentially): p99 ≈ 180ms → Even though each service looks fine individually, the composition degrades. → This is why distributed system SLOs must account for fan-out depth.
▶Walk me through how you'd make a service fault-tolerant at the infrastructure level.
Model Answer
  • Eliminate single points of failure. Any component that exists only once is a SPOF. Load balancers, DB primaries, message brokers — all need a standby or redundant peer.
  • Multi-AZ deployment. Spread instances across at least two availability zones. An AZ failure (power, network) shouldn't take down your service.
  • Health checks + automatic failover. Load balancers stop routing to unhealthy backends. DB primaries need automated replica promotion — manual failover during a 3am incident is a recipe for extended downtime.
  • Graceful degradation via circuit breakers. If the recommendations service goes down, serve without recommendations rather than failing the entire page. Circuit breakers (popularised by Netflix Hystrix) prevent cascading failures — once a downstream service starts failing, stop calling it for a period and return a fallback.
  • Chaos engineering. Deliberately kill instances in production (or staging) to verify failover actually works before you need it. Finding out failover is broken during a real incident is much worse than finding out during a controlled test.
  • Idempotent retries. Ensure operations are safe to retry. Assign idempotency keys and make side effects replayable — the infrastructure will retry failed operations, so your code must handle it.

Interview Questions & Model Answers — click to expand
▶When would you choose DynamoDB over PostgreSQL? What are the trade-offs?
Model Answer
  • Choose DynamoDB when access patterns are simple and predictable (get by partition key, query within a partition), write throughput needs to scale horizontally without operational work, and schema flexibility is genuinely needed. Classic fits: session stores, shopping carts, user preferences, IoT event records.
  • Choose PostgreSQL when you have relational data with many-to-many relationships, need ad-hoc queries (you can't know all access patterns in advance), require ACID transactions across multiple entity types, or your schema evolves frequently in ways that benefit from SQL's flexibility at query time rather than model time.
  • The core trade-off: query flexibility vs operational simplicity. DynamoDB requires you to model your access patterns upfront and encode them into key design. If your access patterns change, you may need a completely new table design. PostgreSQL lets you run any query at any time — but schema changes on large tables are painful, and you manage the operational complexity yourself.
  • Operational overhead. DynamoDB is fully managed — no patching, backup config, replication tuning. PostgreSQL on RDS reduces this, but you still own instance sizing, read replica setup, and failover configuration.
Common mistake: choosing DynamoDB because "it scales better" without modelling access patterns first. DynamoDB is harder to work with once your queries evolve — you can't just add a WHERE clause. Many teams migrate back to Postgres after discovering this.
▶Explain schema-on-read vs schema-on-write. When does each help?
Model Answer
  • Schema-on-write (relational DBs): schema enforced at write time. You can't insert a row that violates the schema. Reads are cheap because the DB guarantees structure. Think: static typing.
  • Schema-on-read (document DBs, data lakes): data written in any shape; the application interprets structure at read time. No migration needed to add a field. Think: dynamic typing.
  • Schema-on-read helps when: different records legitimately have different fields (e.g. a product catalogue where TVs have resolution and shoes have size), or the schema is evolving fast and migration overhead is a genuine bottleneck.
  • Schema-on-write helps when: data quality is critical, multiple teams or systems read the data and need a reliable contract, and you want the DB to catch application bugs at write time — a missing required field fails at insert, not at 2am when a downstream consumer reads it.
Fintech examples: - Payment records → schema-on-write (Postgres). A payment with no amount or currency should never silently persist. Catch it at the DB layer. - User notification preferences → schema-on-read (DynamoDB). A new "dark_mode" preference field can be added to new records without migrating 50M existing ones. Old records just return null.
▶You're building a social network recommendation system. What data model do you use?
Model Answer
  • Graph model. "Friends of friends who like the same things" is a multi-hop graph traversal. Relational DBs can do this with recursive CTEs or self-joins, but performance degrades badly at scale — a 4-hop traversal on a 500M-node social graph takes minutes in SQL.
  • Graph DB (Neo4j / Neptune): stores relationships as first-class citizens. Traversal is O(1) per hop regardless of graph size — the adjacency is stored directly rather than being computed via a join. A query like "find users within 3 hops of User X who bought Product Y" runs in milliseconds.
  • In practice at hyperscale: Facebook and LinkedIn don't use off-the-shelf graph DBs. They built custom graph stores (TAO at Facebook, Voldemort at LinkedIn) optimised for their specific traversal patterns. For a mid-scale system, Neo4j is the correct starting point.
  • Complement with relational: the graph stores relationship topology; user profiles, transactions, and post content stay in Postgres. These are complementary, not mutually exclusive.
CH 03

Storage & RetrievalHow databases actually store and find data — LSM-trees vs B-trees

LSM-Trees write-optimised

Used by: Cassandra, RocksDB, LevelDB, HBase

  • Writes go to in-memory memtable → flushed to immutable SSTables
  • Background compaction merges SSTables; bloom filters speed reads
  • Pros: fast sequential writes, high write throughput
  • Cons: reads may scan multiple SSTables; compaction uses I/O

B-Trees read-optimised

Used by: PostgreSQL, MySQL InnoDB, SQLite

  • Fixed-size pages (~4KB) as a balanced tree; writes update pages in-place
  • WAL (write-ahead log) for crash recovery
  • Pros: predictable O(log n) reads, good range queries
  • Cons: random writes expensive; write amplification

Indexes — Clustered, Secondary, Covering

  • Clustered index: data physically stored in index order. One per table (InnoDB primary key). Fast range scans
  • Secondary index: points to primary key. Requires a second lookup (heap fetch) to get full row
  • Covering index: includes all columns the query needs — no heap fetch required. "Index-only scan" in Postgres EXPLAIN
  • Composite index: column order matters. Left-prefix rule: index on (a,b,c) can serve queries filtering on a, or a+b, or a+b+c — but not b alone

OLTP vs OLAP

  • OLTP: many short reads/writes, latency-sensitive → row-oriented storage (Postgres, MySQL)
  • OLAP: few long analytical queries over millions of rows → column-oriented (Redshift, BigQuery, ClickHouse)
  • Columnar compresses well; only reads the columns a query touches
Interview Questions & Model Answers — click to expand
▶Compare LSM-trees vs B-trees. When would you choose Cassandra over Postgres?
Model Answer
  • LSM-trees are write-optimised. Writes go to an in-memory buffer (memtable) and are flushed sequentially to disk as SSTables. Sequential I/O is far faster than random I/O — LSM-trees handle extremely high write throughput as a result.
  • B-trees are read-optimised. Any key is found in one O(log n) tree traversal. For read-heavy workloads with complex queries, B-trees win. But each write must update a page in place, meaning random I/O.
  • Write amplification. LSM-trees rewrite data during compaction (data written once may be rewritten 10–30x across compaction levels). B-trees rewrite an entire 4KB page for a single-byte change, plus the WAL. LSM-trees typically have lower write amplification for append-heavy workloads.
  • Choose Cassandra when: write throughput dominates, data has a time-series or append-only shape, you need multi-datacenter replication with tunable consistency, and you don't need complex multi-table joins.
  • Choose Postgres when: you need ACID transactions, complex joins, schema enforcement, or strong consistency without distributed coordination overhead.
Real system: fraud detection pipeline - Write path: millions of transaction events/sec → Cassandra (LSM, high write throughput) - Analysis path: fraud analyst runs ad-hoc SQL joins → BigQuery (columnar OLAP) - Operational queries: "show me all transactions for user X" → Postgres (B-tree, fast key lookup) → Three different storage engines, each doing what it's best at.
▶What is write amplification and why does it matter?
Model Answer
  • Definition: write amplification is the ratio of bytes written to disk vs bytes logically written by the application. A factor of 10 means 1KB of application data causes 10KB of disk writes.
  • B-trees: writing one row updates a 4KB page (even for a 10-byte change), writes the WAL entry, and may cascade into page splits that update parent pages. All for a tiny logical write.
  • LSM-trees: compaction repeatedly merges and rewrites SSTables. Data written once might physically be rewritten 10–30x as it moves through compaction levels before reaching the final level.
  • Why it matters: SSDs have finite write endurance (write cycles before failure). High write amplification wears out SSDs faster and directly increases hardware cost at scale. It also saturates disk I/O bandwidth — at high write load, compaction competes with foreground writes and causes latency spikes.
  • Mitigation: tune compaction strategy (STCS vs LCS in Cassandra), use NVMe SSDs over SATA, batch small writes at the application layer.
▶How does a column-oriented database make analytical queries faster?
Model Answer
  • Only read what you need. A query like "average order value by country" touches 2 columns out of potentially 50. Row-oriented storage reads the entire row off disk for every row it scans. Column-oriented storage reads only the columns requested — often 10–50x less data.
  • Better compression. A column stores one type of data — all amounts, all timestamps, all country codes. Repeated values compress extremely well. A column of country codes with 200 unique values across 1B rows compresses to essentially a small dictionary + run-length encoded IDs.
  • CPU cache efficiency. Scanning a column of 64-bit integers is a tight loop that fits in CPU cache lines — modern CPUs can vectorise this. Scanning across rows (mixed types, arbitrary offsets) thrashes the cache.
  • Vectorised processing. BigQuery, DuckDB, and ClickHouse use SIMD CPU instructions to operate on batches of column values simultaneously.
SELECT AVG(amount), country FROM orders GROUP BY country Row-oriented (Postgres): reads all 50 columns for every row, extracts 2 Column-oriented (BigQuery): reads only 'amount' and 'country' columns, compressed On 1 billion rows: - Postgres: ~500GB data read from disk - BigQuery: ~10GB data read (only 2 compressed columns) → BigQuery finishes in seconds; Postgres takes hours
▶What's the difference between a covering index and a secondary index?
Model Answer
  • Secondary index: an index on a non-primary-key column. When used, the DB finds matching rows' primary keys in the index, then performs a second "heap fetch" to retrieve full rows from the main table. Two I/O operations.
  • Covering index: an index that includes all columns the query needs. The DB answers the query entirely from the index — no heap fetch. One I/O operation. Postgres calls this an "Index Only Scan" in EXPLAIN output.
  • When to use covering indexes: high-frequency read queries where eliminating the heap fetch is worth the extra index size. Particularly valuable for OLTP systems under heavy read load.
-- Secondary index only on (user_id): SELECT name, email FROM users WHERE user_id = 123 → Index finds user_id=123 → heap fetch for name and email = 2 I/Os -- Covering index on (user_id, name, email): SELECT name, email FROM users WHERE user_id = 123 → All data is IN the index = 1 I/O, no heap fetch In Postgres EXPLAIN: look for "Index Only Scan" vs "Index Scan" The difference matters enormously at high QPS.

▶Why would you choose Avro over JSON for Kafka messages at Stripe/Wise scale?
Model Answer
  • Size. JSON embeds field names in every message: {"amount":99.99,"currency":"GBP"}. Avro encodes field names once in a schema stored separately; messages contain only binary values. At high volume, this is typically a 3–5x size reduction — meaningful bandwidth and storage savings.
  • Schema evolution with guarantees. Avro has formal rules for backwards/forwards compatibility. Adding a field with a default value is safe. Renaming a field without an alias breaks compatibility. This is a machine-enforceable contract between producer and consumer — JSON gives you no such guarantees.
  • Schema Registry. Confluent Schema Registry stores Avro schemas centrally. Each Kafka message carries a 4-byte schema ID instead of the full schema. Consumers look up the schema to deserialise. This gives you type safety and schema versioning without per-message overhead.
  • Parse speed. Binary parsing is significantly faster than text-based JSON — no string scanning, no UTF-8 decoding per field name.
At 1M messages/sec: - JSON: avg 500 bytes → 500 MB/sec on the wire - Avro: avg 150 bytes → 150 MB/sec on the wire → ~£50K/yr in Kafka storage and egress costs saved → Plus: schema validation catches producer bugs at publish time, not 3am
▶What is schema evolution and how does Protobuf handle it?
Model Answer
  • Schema evolution is the ability to change data schema over time while keeping old and new code interoperable. Critical in distributed systems where you can't redeploy every service simultaneously.
  • Backwards compatibility: new code can read data written by old code. Old code can safely ignore fields it doesn't know about.
  • Forwards compatibility: old code can read data written by new code — it just skips unrecognised fields.
  • Protobuf's mechanism: every field has a number, not just a name. The binary encoding uses field numbers. Adding a new field means assigning it a new number. Old decoders see an unknown field number and skip it (forwards compat). If the new field is optional, old producers simply don't include it (backwards compat).
  • Safe changes: add optional fields, add new message types, rename fields (number unchanged).
  • Unsafe changes: change a field's type, reuse a field number for a different field, remove a required field.
▶What happens if you remove a required field from a Protobuf schema?
Model Answer
  • Backwards compatibility breaks. Old code that still declares the field as required will fail to deserialise messages from new code that omitted it. This is a breaking change.
  • Best practice: never use required in Protobuf 2. Proto3 dropped required entirely — all fields are implicitly optional, sidestepping this class of problem. Mark deprecated fields with a comment rather than removing them, and reserve their field numbers to prevent accidental reuse.
  • Safe removal process: (1) make the field optional everywhere, (2) deploy, (3) stop writing it in a subsequent release, (4) remove from schema while reserving the field number.
// Safe: reserve field number and name to prevent future reuse message Payment { reserved 3; reserved "old_currency_code"; string currency = 4; // new field replacing it }
CH 05

ReplicationKeeping copies of data on multiple machines

Single-Leader most common

  • All writes to leader; replication log sent to followers
  • Sync follower: durable, slower; leader waits for ack
  • Async follower: fast, but can lag; risk of data loss on failover
  • Semi-sync: one sync + rest async — common compromise

Multi-Leader & Leaderless

  • Multi-leader: multiple DCs each accept writes. Conflict resolution is the hard problem (LWW, CRDTs, merge)
  • Leaderless (Dynamo): any node accepts writes. Quorums: w+r>n guarantees overlap. Read repair + anti-entropy sync

Replication Lag Problems

  • Read-your-own-writes: user writes → reads from lagging replica → sees stale data. Fix: route user's own reads to leader for a short window
  • Monotonic reads: user sees data from replica A, then older data from replica B. Fix: sticky sessions to same replica
  • Consistent prefix reads: causally related writes must be seen in order
Interview Questions & Model Answers — click to expand
▶Synchronous vs asynchronous replication — what are the trade-offs?
Model Answer
  • Synchronous replication: the leader waits for at least one follower to confirm it has written the data before acknowledging the write to the client. Guarantees durability — if the leader dies immediately after the write, the follower has it. Cost: write latency is bounded by the slowest synchronous follower, and a follower outage blocks all writes.
  • Asynchronous replication: leader acknowledges immediately and replicates in the background. Lower write latency, but if the leader dies before replicating, that write is lost — committed from the client's perspective, gone from the DB.
  • Semi-synchronous (the practical choice): one follower is synchronous, the rest are async. You have one durable copy at the cost of slightly higher write latency, without being vulnerable to the slowest of N followers.
  • Fintech context: PostgreSQL in AWS RDS Multi-AZ uses synchronous replication to the standby. For payment records, losing a committed transaction is unacceptable. Analytics read replicas can be async — slightly stale aggregate counts are fine.
▶What is replication lag and what problems does it cause? How do you mitigate it?
Model Answer
  • Replication lag is the delay between a write committing on the leader and becoming visible on a follower. Typically milliseconds at low load, but can grow to seconds or even minutes under heavy write pressure.
  • Problem 1 — Read-your-own-writes: a user updates their profile photo, then immediately reloads the page — but the read hits a lagging replica and shows the old photo. Fix: for data the user just modified, route their reads to the leader for a short window (e.g. 30 seconds after the write).
  • Problem 2 — Monotonic reads: user reads from replica A (current), then the next request hits replica B (more lagged) and they see older data — like going back in time. Fix: sticky sessions — always route a given user's reads to the same replica.
  • Problem 3 — Causal consistency: user posts a comment; their friend sees a reply to that comment before they see the comment itself. Fix: ensure causally related reads go to replicas that are equally up-to-date.
▶How does Cassandra handle writes with no single leader? What is a quorum?
Model Answer
  • Leaderless replication. Any Cassandra node can accept a write for any key. The coordinator node (often the receiving node) sends the write to all replica nodes responsible for that key in parallel.
  • Quorum definition: with n replicas, a quorum write requires w nodes to acknowledge. A quorum read requires r nodes to respond. The rule w + r > n guarantees the sets overlap — at least one node you read from was among those that received the write.
  • Common config: n=3, w=2, r=2. Any 2 of 3 replicas must respond. Tolerates 1 node being down. This is Cassandra's QUORUM consistency level.
  • Trade-off: lower w and r (e.g. w=1, r=1) gives faster, more available operations at the cost of consistency. Higher values (w=3, r=3 — ALL) give maximum consistency but any single node failure blocks all operations.
n=3 replicas, w=2, r=2: Write: send to nodes A, B, C. A and B ack → write succeeds (C may lag) Read: query A and B. A returns v2 (latest), B returns v1 (stale) → return v2 + repair: Cassandra updates B with v2 in background (read repair) Guarantee: w+r=4 > n=3, so read set and write set always overlap by at least 1 node. You will always read at least one node that has the latest write.
▶How would you handle a leader failover to avoid split-brain or lost writes?
Model Answer
  • Split-brain occurs when two nodes both believe they are the leader and accept writes independently. Merging those diverged write logs later is extremely hard and often causes data loss.
  • Prevention — fencing tokens: the lock/lease service issues a monotonically increasing token with each leader grant. Storage layer rejects writes from a node whose token is older than the current leader's. A former leader resuming after a network partition can't corrupt data — its token is stale.
  • Prevention — epoch numbers (Raft/Paxos): each leader term has an epoch number. Messages from old-epoch leaders are rejected by followers.
  • Handling lost writes: with async replication, if the leader dies before replicating, those writes are gone. Accept this by using async only for non-critical data, or use synchronous replication to at least one follower for anything that can't be lost.
  • Practical failover steps: detect failure via health check timeout → elect new leader via consensus (Raft) or operator → configure all followers to replicate from new leader → if old leader returns, force it to become a follower, not attempt re-election.
CH 06

Partitioning (Sharding)Splitting data across nodes to scale beyond a single machine

Key-Range Partitioning range queries

  • Partition by key range: A–F on shard 1, G–M on shard 2
  • Pro: efficient range queries — scan one partition
  • Con: hot spots — time-series data all lands on latest shard
  • Used by: HBase, BigTable, MongoDB range sharding

Hash Partitioning even distribution

  • Hash the key → assign to partition. Even load distribution
  • Con: range queries must hit all partitions (scatter-gather)
  • Celebrity hot-spot: salt the key, aggregate on read
  • Cassandra: hash on first key part, range on second — best of both

Consistent Hashing & Rebalancing

  • Consistent hashing: nodes on a circular ring. Add/remove node → only adjacent partitions move. Minimises data movement vs hash mod N
  • Virtual nodes (vnodes): each physical node owns many ring positions → smoother rebalancing, better load distribution
  • Fixed partitions: create 1000 logical partitions upfront, move whole partitions between nodes (Elasticsearch, Riak)
  • Dynamic partitions: split when too large, merge when too small (HBase)
Interview Questions & Model Answers — click to expand
▶How would you shard a user table across 10 DB nodes? Range vs hash trade-off?
Model Answer
  • Hash sharding on user_id. Hash(user_id) mod 10 assigns each user to a shard evenly. The vast majority of user lookups are by user_id, so reads hit exactly one shard — optimal.
  • Range sharding is wrong here. If you range-shard on user_id (1–1M on shard 1, etc.), new registrations all land on the highest shard — a write hot spot. Range-sharding on username alphabetically gives wildly uneven distribution (S–T has far more users than X–Z).
  • The cross-shard join problem. "Find all users who joined in the last 7 days" must hit all 10 shards and merge results — scatter-gather. Acceptable if rare. For frequent range queries, maintain a separate global secondary index partitioned by created_at, or use a read-only analytics copy.
  • Rebalancing to 11 nodes. Hash mod 10 → hash mod 11 means nearly every user moves. This is why you should use consistent hashing or fixed-partition mapping (1000 logical partitions → 10 physical nodes; adding node 11 moves only ~9% of logical partitions).
▶One shard is receiving 80% of writes. What's causing it and how do you fix it?
Model Answer
  • Diagnosis: hot key. A single key (or small set of keys) receiving disproportionate traffic. Classic examples: a celebrity user's post going viral (millions of interactions on one record), a trending hashtag, or a shared counter (total likes on a viral post).
  • Fix 1 — Key salting: append a random suffix (0–9) to the hot key, spreading it across 10 sub-partitions. Write to celebrity_123_3 (random). On read, query all 10 suffixes and aggregate. Distributes write load; adds read aggregation cost.
  • Fix 2 — In-memory buffering for counters: if the hot writes are increments (like counts), buffer them in Redis and flush to the DB periodically. Reduces DB write load by orders of magnitude — Redis is single-threaded but extremely fast for atomic increments.
  • Fix 3 — Separate partition strategy for hot entities. Identify entities that will always be hot (celebrity accounts, top-level config records) and give them dedicated storage outside the normal sharding scheme.
  • Monitoring: per-partition write QPS should be in your observability stack. Alert on any shard receiving >3x average write rate.
Example: Twitter's celebrity tweet problem - Normal users: fan-out on write (pre-compute followers' timelines at write time) - Celebrities (>1M followers): fan-out on read (compute timeline at read time from celebrity's posts) → Same root cause (hot key), different solution for different access patterns
▶What is consistent hashing and why do distributed systems use it?
Model Answer
  • The problem with hash mod N: with 10 nodes and hash mod 10, adding an 11th means almost every key remaps to a different node — massive data movement to rebalance.
  • Consistent hashing solution: arrange both nodes and keys on a circular hash ring (0 to 2³²). Each key is assigned to the first node clockwise from its position. Adding a node only affects keys between it and its predecessor — on average 1/N of all keys move.
  • Virtual nodes solve uneven distribution: nodes placed randomly on the ring land unevenly — some nodes get large arc ranges, others tiny ones. Virtual nodes give each physical node 100–200 positions on the ring. Adding/removing a node redistributes load from many small segments rather than one large arc. Cassandra uses 256 vnodes by default.
Without consistent hashing (10 → 11 nodes, hash mod N): ~91% of keys need to move — almost total data reshuffle With consistent hashing (1000 vnodes, 10 → 11 nodes): ~9% of keys move — only the vnodes re-assigned to the new node → This is why Cassandra can add a node with minimal disruption
▶How do secondary indexes work across partitions?
Model Answer
  • Local (document-based) secondary indexes: each partition independently indexes its own data. A query on the indexed field must hit all partitions and merge results — scatter-gather. Simple to write to (index update is local), expensive to read from. Used by MongoDB, Cassandra, Elasticsearch.
  • Global (term-based) secondary indexes: the index itself is partitioned separately from the main data. A read on the index hits one partition of the index (which contains all matching records from all main data partitions). Reads are fast. Writes are expensive — a single write to the main data may need to update index partitions on different nodes, requiring distributed coordination. Used by DynamoDB GSIs.
  • Trade-off: local indexes are write-friendly, read-expensive (scatter-gather). Global indexes are read-friendly, write-expensive (cross-partition updates). Choose based on your read/write ratio for indexed queries.

TransactionsACID, isolation levels, and concurrency anomalies — critical for fintech

ACID — what each property actually means

▶Explain ACID. What does isolation actually mean in practice?
Model Answer
  • Atomicity — a transaction is all or nothing. If you debit £100 from account A and the credit to account B fails, the debit is rolled back automatically. No partial state persists. Implemented via WAL — on crash, incomplete transactions are rolled back on recovery.
  • Consistency — the odd one out. It's defined by the application, not the database. The DB enforces constraints (foreign keys, CHECK constraints), but the broader invariant — "total money in the system never changes" — is something you encode in your transaction logic. The DB is a tool for preserving it.
  • Isolation — in practice, it means: if transaction A and B run concurrently, the result is as if they ran one at a time in some order. Full isolation (serializable) is expensive, so most DBs default to weaker levels. PostgreSQL defaults to Read Committed — prevents dirty reads, but still allows non-repeatable reads and write skew. Many developers assume this is "safe enough" for financial transactions. It isn't for all cases.
  • Durability — committed transactions survive crashes. Write-ahead logging (WAL) writes the log to durable storage before updating in-memory pages. On crash, the WAL is replayed to reconstruct committed state.
▶What is write skew? Give a real example. How do you prevent it?
Model Answer
  • Write skew definition: two transactions both read overlapping data, each independently decides to write based on what they read, and the combined result violates an invariant that each individual write would have preserved.
  • Classic example — on-call doctors: hospital rule: at least 1 doctor must be on-call. Dr A checks: 2 on-call (A and B). Dr B checks: 2 on-call. Both independently decide it's safe to go off-call. Both commit. Now 0 doctors on-call. Neither transaction did anything wrong in isolation — together they broke the invariant.
  • Fintech example — balance check before withdrawal: two concurrent requests both read balance=£500, both decide £400 is safe to withdraw, both commit. Account is now -£300.
  • Prevention options:
    • Serializable Isolation (SSI): PostgreSQL's SSI detects this conflict pattern at commit time and aborts one transaction with a serialization failure. Application retries. Correct, but adds retry logic.
    • SELECT FOR UPDATE: explicitly lock the rows you read as part of your decision. Other transactions block until yours commits. Correct and simple, but reduces concurrency.
    • Materialise the conflict: create a lock row representing the contested resource (e.g. one row per time slot for meeting rooms) so transactions can SELECT FOR UPDATE on it explicitly.
Critical nuance: Snapshot Isolation (Postgres's "Repeatable Read") does NOT prevent write skew. You must use Serializable isolation or explicit locking. Many developers don't know this and assume Repeatable Read is safe.
▶What is the difference between Snapshot Isolation and Serializable Isolation?
Model Answer
  • Snapshot Isolation (SI): each transaction reads from a consistent snapshot of the DB taken at transaction start. Reads don't block writes; writes don't block reads. Prevents dirty reads and non-repeatable reads. But doesn't prevent write skew — two transactions can both read the same snapshot and make conflicting decisions that together violate an invariant.
  • Serializable Snapshot Isolation (SSI): runs under snapshot isolation but additionally tracks read/write dependencies between transactions. At commit time, it detects dependency cycles — the pattern that produces write skew — and aborts one conflicting transaction. The serialization failure is returned as an error; the application must catch and retry.
  • Practical difference: SI has higher throughput (no aborts) with weaker guarantees. SSI prevents all anomalies at slightly lower throughput due to retries. Under low contention, SSI abort rates are very low — the overhead is minimal in practice.
  • PostgreSQL specifics: Postgres "Repeatable Read" = Snapshot Isolation. Postgres "Serializable" = SSI (since v9.1). This naming confuses many engineers — "Repeatable Read" sounds strong but doesn't prevent write skew.
▶Why is 2PC problematic in distributed systems? What are the alternatives?
Model Answer
  • Two-Phase Commit: coordinator sends "prepare" to all participants → all vote yes/no → coordinator sends "commit" or "abort". Works when everyone is available.
  • The core problem — coordinator failure: if the coordinator crashes after sending "prepare" but before sending "commit", all participants are stuck in a prepared state. They hold locks. They cannot safely commit or abort without knowing the coordinator's decision. This is the "in-doubt transaction" — locks held, progress blocked, until the coordinator recovers. Recovery time = minutes to hours in a bad failure scenario.
  • Alternative 1 — Saga pattern: decompose the distributed transaction into a sequence of local transactions, each with a compensating action for rollback. No distributed locks. No coordinator single point of failure. Trade-off: intermediate states are briefly visible — eventual consistency.
  • Alternative 2 — Outbox pattern: write to local DB and an outbox table in the same ACID transaction. A separate relay process reads the outbox and publishes to Kafka. Guarantees exactly-once event publishing without any distributed coordination.
Saga example (hotel + flight booking): Step 1: Book flight → local tx succeeds Step 2: Charge card → local tx succeeds Step 3: Book hotel → FAILS (no rooms) Compensate Step 2: Refund card (compensating transaction) Compensate Step 1: Cancel flight (compensating transaction) → No distributed lock held at any point → Intermediate state (flight booked, card charged) briefly visible → Application must handle compensation failures too (retry with idempotency)
▶How do you handle a payment that needs to debit one service and credit another?
Model Answer
  • Same DB — use a single ACID transaction. Debit and credit in one BEGIN/COMMIT. If either fails, both roll back. Simple, correct, fast. This is always the first choice when possible.
  • Different services/DBs — use Saga + Outbox:
    • Service A: debit account A + write "DebitSucceeded" event to local outbox — single ACID tx
    • Outbox relay: publishes event to Kafka (with at-least-once delivery and retries)
    • Service B: consumes event, credits account B — single ACID tx with idempotency key
    • If B fails repeatedly: compensating event triggers credit back to A
  • Idempotency is non-negotiable. Kafka delivers at-least-once. Service B must not credit twice for the same event. Use the event ID as an idempotency key with a unique DB constraint — duplicate inserts fail gracefully.
  • What not to do: write to DB and publish to Kafka in the same application request without the Outbox pattern. If the DB write succeeds and the Kafka publish fails, the debit has no corresponding credit — money disappears.

The Trouble with Distributed SystemsPartial failures, unreliable networks, and clocks

Partial Failures

In distributed systems, a node can be slow, a packet lost, or a node appears dead from one side and alive from another. You cannot know whether a remote operation succeeded.

  • Timeouts: only way to detect dead nodes — but a timed-out request may have succeeded
  • Idempotency: the only safe response to uncertainty

Unreliable Clocks

  • Wall clock: can jump backwards (NTP). Never use to order events across nodes
  • Last-write-wins with wall clocks: silently drops data when clocks disagree (Cassandra default)
  • Logical clocks (Lamport): counters capturing causal ordering — safe for event ordering
  • Google TrueTime: returns [earliest, latest]. Spanner waits out the uncertainty window

Process Pauses & Fencing Tokens

A process holding a lease can pause (GC, VM migration), let the lease expire, resume — and still think it holds the lock. Solution: fencing tokens. Each lock grant comes with a monotonically increasing token. Storage rejects writes with older tokens. Even a paused process resuming with a stale token can't corrupt data.

Interview Questions & Model Answers — click to expand
▶Why can't you rely on wall clock timestamps to order events across distributed nodes?
Model Answer
  • Clocks drift and are corrected. Each server's hardware clock drifts relative to true time. NTP corrects this by stepping the clock backward or forward. If Node A's clock is 100ms fast and NTP corrects it, an event timestamped at T=1000ms gets reassigned to T=900ms — now it appears to have happened before events on Node B that it actually followed.
  • Last-write-wins becomes data loss. If you use wall clock timestamps to pick the "latest" write in conflict resolution, you're trusting clock synchronisation — which is never perfect. Cassandra's LWW mode can silently drop writes from nodes with slightly slow clocks. There's no error, no warning. The data just doesn't exist.
  • What to use instead: logical clocks. Lamport timestamps assign a counter that increments on every event and synchronises (takes the max + 1) when nodes communicate. They capture causal ordering without requiring synchronised physical clocks. Vector clocks extend this to detect concurrent vs causally related events.
Node A (clock 200ms fast): writes key X, timestamp=1200ms Node B (correct clock): writes key X, timestamp=1100ms (happens AFTER A's write) LWW picks A's write (1200 > 1100) even though B's write was causally later. B's write is silently dropped. No error returned to the client. Data loss. With logical clocks (Lamport): A writes event at counter=5, sends to B B receives counter=5, increments to 6 before writing → B's event is provably after A's regardless of wall clock
▶What is a fencing token and why is it needed for distributed locks?
Model Answer
  • The problem: a process acquires a distributed lock with a 30-second lease, then pauses (JVM GC stop-the-world, Linux scheduler, network stall). The lease expires. Another process acquires the lock and starts writing. The first process resumes, still believes it holds the lock, and writes — now two processes have concurrent exclusive access. Data corruption.
  • Fencing token solution: every time the lock service grants a lock, it issues a monotonically increasing token number (e.g. via Redis INCR or ZooKeeper's zxid). The lock holder must include this token in every write to the protected resource. The storage layer tracks the highest token it's seen and rejects any write with a lower token.
  • Why this works: when process 1 pauses and process 2 acquires the lock, process 2 gets token=34. If process 1 resumes and tries to write with token=33, the storage rejects it (33 < 34). Process 1 gets an error, learns it lost the lock, and fails gracefully instead of corrupting data.
  • Critical requirement: the storage layer must enforce the check. A lock service alone isn't sufficient — if the storage accepts any write regardless of token, fencing doesn't help. ZooKeeper's monotonic zxids are commonly used as fencing tokens.
Without fencing: Client 1: acquires lock, pauses for 35s (GC) Client 2: lock expires, acquires lock, writes data (v2) Client 1: resumes, thinks it has lock, writes data (v1) → overwrites v2 → corruption With fencing tokens: Client 1: acquires lock, gets token=33, pauses Client 2: acquires lock, gets token=34, writes with token=34 → storage accepts Client 1: resumes, tries to write with token=33 → storage sees 33 < 34 → REJECTS → Client 1 gets an error, handles gracefully. No corruption.
▶A network request times out. How do you know if the operation actually happened? How do you handle it?
Model Answer
  • You can't know from the timeout alone. The request might have been: (1) lost before reaching the server — not processed, (2) received and processed but the response lost in transit — processed, (3) received but server is slow — may or may not process it. These three cases look identical to the caller.
  • Design for this with idempotency keys. Before sending any non-idempotent request, generate a unique UUID idempotency key. Include it in the request. The server processes it once and stores the result keyed to the idempotency key. On retry with the same key, the server returns the stored result — no reprocessing. Now you can retry indefinitely until you get a response.
  • Query for status first. For operations with side effects (a payment, a booking), implement a status endpoint: GET /payments/{idempotency_key}. On timeout, query status before retrying. If it succeeded, return the existing result. If not found, retry. This avoids double-processing even without idempotency server support.
  • This is how Stripe works. Every POST requires an Idempotency-Key header. They store the response for 24 hours per key. The client retries with the same key until it gets a definitive answer — success or failure.

Consistency & ConsensusCAP theorem, linearizability, and Raft

CAP Theorem

SystemChoice under network partition
ZooKeeper, etcd, HBaseCP — stops accepting writes if quorum unreachable. Consistent but may be unavailable
Cassandra, DynamoDB, CouchDBAP — always accepts writes. Eventually consistent

Note: "Consistency" in CAP means linearizability specifically — not ACID consistency or eventual consistency. Widely misunderstood.

Linearizability

System appears to have a single atomic register. Every read returns the most recently written value. Required for: leader election, distributed locks, uniqueness constraints, financial balances. Cost: coordination on every operation → high latency.

Raft

Leader election + log replication + safety guarantees. Used by etcd, CockroachDB, TiKV, Consul. Committed log entries survive leader failures. Leader elected by majority vote from nodes with most up-to-date log.

Interview Questions & Model Answers — click to expand
▶Explain CAP theorem. Where does Cassandra sit vs ZooKeeper?
Model Answer
  • CAP states: in a distributed system that experiences a network Partition, you must choose between Consistency (linearizability — every read sees the most recent write) and Availability (every request receives a response, even if it may not reflect the latest write). You can't guarantee both simultaneously when nodes can't communicate.
  • Critical nuance — "Consistency" in CAP means linearizability. Not ACID consistency. Not eventual consistency. Linearizability: after a write completes, any subsequent read from any node must return that value or a later one. This is a very specific, strong definition.
  • Cassandra (AP): always accepts reads and writes, even during a partition. Nodes that are isolated from each other continue operating. After the partition heals, they reconcile via anti-entropy and read repair. You might read stale data during a partition — that's the trade-off. Appropriate for: shopping carts, user activity feeds, sensor data.
  • ZooKeeper (CP): if ZooKeeper loses quorum (more than half of nodes unreachable), it stops accepting writes entirely — returns errors rather than potentially stale answers. Appropriate for: leader election (a stale "you are the leader" response is worse than an error), distributed locks, cluster configuration.
  • Practical framing: network partitions between well-connected DCs are rare. Most of the time both systems operate fine. The CAP choice is really about failure mode design — what should the system do when things go wrong?
▶What is linearizability? When do you need it vs when can you accept eventual consistency?
Model Answer
  • Linearizability means the system behaves as if there is one copy of the data and all operations execute atomically in real time. Once a write completes, any read from any node returns that value or a later one. No stale reads ever.
  • You need linearizability when:
    • Distributed locks — only one process can hold a lock at a time; a stale read that says "lock is free" when it isn't causes correctness failures
    • Leader election — exactly one node should believe it's the leader
    • Uniqueness constraints — a username taken by exactly one account
    • Account balance check before a debit — must see latest balance, not a stale read
  • Eventual consistency is fine when:
    • User profile display — seeing a slightly stale bio for 2 seconds is harmless
    • Product catalogue — a 5-second-stale price is acceptable in most e-commerce contexts
    • Social media like counts — approximate is fine; nobody notices ±10 likes
    • Search indexes — indexing lag of seconds to minutes is widely acceptable
  • Cost of linearizability: requires synchronous coordination (cross-node roundtrips) on every operation. Adds latency. Reduces availability (system must decline requests if it can't confirm with a quorum). Choose it deliberately, not by default.
▶How does Raft achieve consensus? What happens when the leader fails?
Model Answer
  • Three sub-problems: leader election, log replication, and safety (only one leader per term; committed entries are never lost).
  • Leader election: all nodes start as followers. If a follower doesn't receive a heartbeat within the election timeout (randomised 150–300ms), it becomes a candidate, increments its term, and sends RequestVote RPCs. A node wins by receiving votes from a majority. Voting rule: a node only votes for a candidate whose log is at least as up-to-date as its own — this ensures the elected leader always has all committed entries.
  • Log replication: the leader receives client requests, appends them to its local log, and sends AppendEntries RPCs to all followers in parallel. Once a majority acknowledge the entry, the leader marks it committed, applies it to its state machine, and responds to the client. Committed entries are durable — they survive any subsequent leader failure.
  • When the leader fails: followers stop receiving heartbeats. After the election timeout, a follower starts a new election. The first to reach majority wins. The voting rule ensures the new leader has all committed entries. Uncommitted entries from the old leader may be overwritten — safe because clients haven't received a success response for them. Typical failover time in etcd: <500ms.
3-node cluster (A=leader, B, C): 1. A receives write W, appends to log[5] 2. A sends AppendEntries to B and C 3. B acks. C is slow. A commits (majority = 2/3). Responds to client. 4. A crashes before C receives the entry New election: - B or C wins (both have committed entry at log[5] — B does, C doesn't) - B must win (B's log is more up-to-date → gets vote from C → majority) - C receives missing entry from B as new leader → cluster converges
▶How would you implement distributed leader election without ZooKeeper?
Model Answer
  • Option 1 — Raft library: use hashicorp/raft (Go), etcd's raft package, or implement via etcd directly (which runs Raft internally and exposes primitives). Heavyweight but provably correct. The right answer for a new distributed system.
  • Option 2 — Redis SET NX + fencing: use SET lock_key unique_token NX PX 30000 (set if not exists, 30s expiry). First node to succeed wins. Renew before expiry to maintain leadership. Add a fencing token (Redis INCR on a separate key) so storage can reject writes from stale leaders. Limitation: Redis as a single node is itself a SPOF. Redlock (multi-node Redis) reduces this but has known edge cases under clock skew.
  • Option 3 — Database row lock: use a single DB row as the election record. Leader continuously updates a heartbeat timestamp. Other nodes watch: if heartbeat goes stale (leader crashed), attempt to claim leadership via compare-and-swap (UPDATE ... WHERE current_leader = NULL or heartbeat_old). Works well when you already depend on a strongly consistent DB.
  • Honest recommendation: use etcd or Consul. They implement Raft correctly. Building distributed consensus from scratch is extremely error-prone — even the Raft paper has subtleties that most implementations get wrong.

▶Explain MapReduce. How would you count unique users per day across 1TB of logs?
Model Answer
  • MapReduce overview: the framework splits input across mappers running in parallel. Each mapper processes its chunk and emits (key, value) pairs. The framework shuffles and sorts by key, grouping all values for the same key together. Reducers receive all values for each key and produce output.
  • For unique users per day:
    • Map phase: parse each log line, emit (date, user_id)
    • Shuffle: groups all (date, user_id) pairs by date
    • Reduce phase: for each date, insert user_ids into a set, emit (date, set.size())
  • Scale concern — memory in the reducer: a busy day might have 100M unique user_ids. At 8 bytes each, that's 800MB per reducer — manageable but tight. For extreme scale, use HyperLogLog in the reducer: a probabilistic data structure that estimates distinct count with ~2% error using kilobytes of memory.
  • In Spark: df.groupBy("date").agg(countDistinct("user_id")) handles this in one statement. Spark's in-memory shuffle is orders of magnitude faster than MapReduce's HDFS-based shuffle for this type of aggregation.
Spark implementation: logs_df .groupBy("date") .agg(countDistinct("user_id").alias("unique_users")) .orderBy("date") .write.parquet("s3://output/unique-users/") For 1TB of logs at 1KB/line = ~1B log lines Spark on 20 nodes (each 32GB RAM): ~5 minutes MapReduce equivalent: ~45 minutes (HDFS reads/writes between stages)
▶Why is Spark faster than MapReduce? What are the trade-offs?
Model Answer
  • In-memory intermediates. MapReduce writes every stage's output to HDFS before the next stage can start — disk reads and writes between every step. Spark keeps data in memory across stages as RDDs/DataFrames. Disk I/O is the bottleneck in most batch jobs; eliminating it gives 10–100x speedup for multi-stage jobs.
  • DAG execution with query optimisation. Spark models a job as a directed acyclic graph and optimises the whole plan before execution (Catalyst query optimiser). MapReduce is a fixed two-stage model — a 10-stage job needs 10 separate MapReduce jobs, each reading and writing HDFS.
  • Lazy evaluation. Spark transformations build a plan but don't execute until an action is called. This lets the optimiser push down filters, eliminate unnecessary columns, and merge stages before any data moves.
  • Trade-offs:
    • Memory pressure: if data doesn't fit in the cluster's RAM, Spark spills to disk and performance degrades significantly — sometimes worse than MapReduce
    • Fault recovery is more expensive: rebuilding a lost RDD partition requires replaying the full transformation lineage. For long pipelines with many transformations, this can take longer than MapReduce's simpler re-run-from-HDFS approach
    • MapReduce is more predictable for very large shuffles — HDFS materialisation makes failures cheap to recover from
▶What is a broadcast join and when would you use it?
Model Answer
  • Standard shuffle join: both tables are shuffled so matching keys land on the same reducer node. This can involve sending terabytes of data across the network — the shuffle is typically the most expensive operation in a batch job.
  • Broadcast join: if one table is small enough to fit in memory (typically <10MB by Spark default, configurable), copy it to every worker node. Each worker joins its partition of the large table against the local in-memory copy. Zero network shuffle for the large table.
  • When to use it: joining a large fact table with a small dimension table. Classic example: joining 1TB of payment records (fact) with a 50KB currency lookup table (dimension). Without broadcast: 1TB shuffled. With broadcast: 50KB sent to 100 workers, 1TB stays in place.
Spark — explicit broadcast hint: from pyspark.sql.functions import broadcast payments_df.join( broadcast(currency_df), # small table → broadcast to all workers on="currency_code" ) → Spark skips the shuffle entirely for currency_df → Instead of: (1TB payments shuffled) + (50KB currency shuffled) → Just: (1TB payments read locally) + (50KB replicated to 100 workers) → Eliminates the most expensive operation in the join
▶Compare Lambda vs Kappa architecture for a real-time + historical reporting system.
Model Answer
  • Lambda architecture: two parallel pipelines. Batch layer reprocesses all historical data nightly for accuracy. Speed layer processes the last few hours in near-real-time for freshness. A serving/query layer merges results from both. The problem: you maintain two separate codebases for the same business logic — a batch version (Spark) and a streaming version (Flink/Kafka Streams) — and keeping them in sync over time is a constant maintenance burden.
  • Kappa architecture: only a streaming pipeline. All data flows through Kafka. The stream processor handles real-time. For historical reprocessing, replay Kafka from offset 0 through a new version of the streaming job. One codebase. Simpler to maintain, test, and reason about.
  • When Lambda still makes sense: if batch processing uses fundamentally different algorithms from streaming (e.g. complex iterative ML training that can't be expressed as a streaming computation), Lambda is pragmatic. Or if your Kafka retention is short and you need a separate historical store that a batch job reads from.
  • Modern default: most new systems choose Kappa. Kafka's configurable long retention (days to weeks) makes historical replay feasible. Flink is mature enough to handle batch-like aggregations as streaming jobs. LinkedIn, Uber, and most large-scale data platforms have moved to Kappa.

Kafka Architecture

Producer → Topic: payments (partitioned across brokers) ├── Partition 0: [offset 0][offset 1][offset 2]... └── Partition 1: [offset 0][offset 1]... ↓ Consumer Group A (fraud detection) — each partition → one consumer Consumer Group B (analytics) — independent offset, replays freely

Ordering guaranteed within a partition. Retention enables replay. Consumers track their own offsets — Kafka is agnostic to consumer progress.

Delivery Semantics

  • At-most-once: fire-and-forget. May lose messages. Rarely acceptable
  • At-least-once: retry on failure, may duplicate. Consumer must be idempotent
  • Exactly-once: idempotent producer + transactional API. Correct but expensive

Windowing

  • Tumbling: fixed non-overlapping windows ("sales per 5 min")
  • Sliding: overlapping ("avg over last 5 min, updated every 1 min")
  • Session: grouped by activity gap ("all events within 30 min of each other")
  • Watermarks: track event time lag; decide when to safely close a window
Interview Questions & Model Answers — click to expand
▶How does Kafka guarantee ordering? What happens with multiple partitions?
Model Answer
  • Ordering guaranteed within a partition only. Kafka appends messages to a partition's log in the order they're received. Consumers read in offset order — strict FIFO within a partition.
  • No ordering guarantee across partitions. A topic with 10 partitions can have messages from different partitions consumed in any relative order. This is intentional — it's what enables parallel consumption.
  • Ensuring related messages land on the same partition: use a message key. Kafka hashes the key to determine partition assignment. All messages with the same key (e.g. customer_id, order_id) land on the same partition in insertion order. This guarantees sequential processing for all events belonging to one entity.
  • Practical implication: for a payment system, partition by customer_id. All events for customer 123 are on partition 5, in order. A debit event always precedes the notification for that debit because they're sequentially ordered on the same partition.
producer.send( topic="payments", key=customer_id, # ← determines partition assignment value=payment_event # ← payload ) All events for customer_id="abc123" → always go to the same partition → Guaranteed ordering for that customer's event stream → Other customers process in parallel on different partitions
▶At-least-once vs exactly-once delivery. Which for a payments system and why?
Model Answer
  • At-least-once: the producer retries until it gets an ack. If the broker received the message but the ack was lost in transit, the producer retries and the message is delivered twice. Consumer must handle duplicates.
  • Kafka exactly-once: idempotent producer assigns a per-partition sequence number — broker deduplicates retries within a session. Transactional API wraps produce + consumer offset commit in an atomic operation. Consumers set isolation.level=read_committed to see only committed messages.
  • Pragmatic recommendation for payments: at-least-once + idempotent consumer. Reasoning: (1) Kafka's exactly-once only covers the Kafka-to-Kafka path — your DB write still needs idempotency, so you need it anyway; (2) idempotent consumers are easier to reason about, test, and debug; (3) Kafka transactions add non-trivial throughput overhead and latency.
  • Implementing idempotent consumer: include a unique event_id in each Kafka message. Consumer uses it as an idempotency key with a unique DB constraint. On duplicate message, the duplicate insert fails with a constraint violation — catch it and treat as success.
Common misconception: "exactly-once" sounds perfect for payments. But true end-to-end exactly-once requires coordination at every layer — Kafka transactions, idempotent DB writes, idempotent PSP calls. The complexity is rarely worth it over at-least-once + idempotency. Stripe uses at-least-once + idempotency keys.
▶What is a watermark in stream processing? Why is it needed?
Model Answer
  • The problem: events arrive out of order. A mobile app might buffer events offline for 2 hours before reconnecting. A click at 14:05 might arrive at 14:23. If you're computing a 5-minute tumbling window 14:00–14:05, when do you close it? If you close immediately at 14:05 (processing time), you miss late events. If you wait forever for stragglers, results are never emitted.
  • Watermark: a declaration that "all events with timestamp < T have now arrived." Advances over time as events flow through the stream. When the watermark passes a window's end time, the window closes and results are emitted — with a configurable grace period for late arrivals.
  • How Flink uses it: Flink tracks max event timestamp seen minus an "allowed lateness" (e.g. 5 minutes). The watermark advances as events arrive. When watermark > window_end, window closes.
  • Late event handling: events arriving after the watermark has passed their window can be dropped, sent to a side output for separate handling, or used to emit a correction to the already-computed result.

▶How would you build a real-time fraud detection system using Kafka and Flink?
Model Answer
  • Event ingestion: every transaction event published to a Kafka topic, partitioned by customer_id — all events for one customer arrive ordered on the same partition.
  • Flink stateful processing: a Flink job consumes from Kafka. For each customer_id (the key), Flink maintains keyed state: recent transaction history, spend velocity per merchant category, geographic location sequence. State lives in Flink's managed memory, checkpointed to S3 every 30 seconds for fault tolerance.
  • Rule evaluation: for each incoming transaction, evaluate rules against keyed state: "3+ transactions in different countries within 60 minutes", "amount 10x above 30-day average", "two geographically impossible locations within 10 minutes" (velocity check). Rules implemented as Flink ProcessFunction operators with access to the full state.
  • Windowed aggregations: sliding windows (last 1 hour, updated per event) on spend per merchant category, transaction velocity, etc. Feed these as features into rule evaluation and downstream ML scoring.
  • Output: fraud signals published to a separate Kafka topic. Downstream decision service applies ML model scores and triggers: block, flag for review, or pass. All decisions persisted to DB for audit.
  • Latency requirement: sub-100ms end-to-end from transaction event to fraud decision, before the payment authorisation request times out at the card network.
▶What is CDC? Why is it better than dual-writes for syncing a DB with a message queue?
Model Answer
  • CDC (Change Data Capture) reads a database's write-ahead log and converts every committed insert/update/delete into a stream of events, published to Kafka. The event is emitted if and only if the DB transaction committed — atomicity guaranteed by the WAL.
  • Dual-write problem: your application writes to DB, then publishes to Kafka. If the DB write succeeds and the Kafka publish fails (network timeout, broker down, application crash), you have a DB record with no corresponding event. Elasticsearch, Redis, and analytics DBs that consume from Kafka now diverge from the source of truth. No automatic recovery — you'd need manual reconciliation to find and republish missed events.
  • Why CDC is better: the DB commit is the single source of truth. Debezium reads Postgres's logical replication slot after commit — if it's in the WAL, it committed. Debezium publishes to Kafka with at-least-once delivery; consumers handle duplicates with idempotency keys. Adding a new consumer requires zero changes to the application.
Without CDC (dual-write): 1. app.db.insert(order) ← succeeds 2. kafka.publish(order) ← FAILS (broker down) → DB has order. Elasticsearch doesn't. Search is now wrong. → No automatic detection or recovery. With CDC (Debezium → Kafka): 1. app.db.insert(order) ← succeeds, WAL entry written 2. Debezium reads WAL ← sees committed insert 3. Debezium publishes ← retries until Kafka acks → Guaranteed: if it's in Postgres, it will be in Kafka. → Elasticsearch, Redis, BigQuery all consume independently from Kafka.

The Future of Data SystemsUnbundling databases, event logs, and end-to-end correctness

Unbundling the Database

Modern systems decompose the monolithic DB into specialised components connected by event streams. Postgres for transactions → CDC → Kafka → Elasticsearch, Redis, BigQuery. Each does one thing well. The event log is the system of record; derived stores are views that can always be rebuilt.

Immutable Event Log

Any derived data (cache, search index, ML feature store) should be reproducible by replaying the source event log. Operations must be idempotent. This gives you the ultimate debuggability: rebuild any downstream system from scratch, and reconstruct historical state at any point in time.

▶If you were designing a data platform from scratch for a fintech at scale, how would you structure the data flow?
Model Answer
  • Transactional layer — PostgreSQL. All financial records: payments, ledger entries, account balances. ACID transactions, strong consistency, append-only ledger pattern. For global distribution, CockroachDB (distributed Postgres-compatible with serializable isolation).
  • Event streaming backbone — Kafka. All DB changes flow via CDC (Debezium) into Kafka. This becomes the central nervous system. All derived data is computed from this stream. Decouples producers from consumers; enables replay for new consumers and reprocessing.
  • Derived stores:
    • Redis: account balance cache for fast reads, session store, rate limiter counters, idempotency key store
    • Elasticsearch: transaction search, customer lookup, support queries ("find all transactions for this card last 30 days")
    • BigQuery: historical analytics, regulatory reporting, ML feature store
  • Real-time processing — Flink. Fraud detection, AML monitoring, balance aggregation, real-time dashboards. Stateful operators over Kafka streams. Checkpointed to S3 for fault tolerance.
  • Batch processing — Spark/dbt. Nightly reconciliation (internal ledger vs PSP settlement files), regulatory reports, ML model training on historical data, data quality checks.
  • Invariant: Postgres is always the source of truth. If Redis and Elasticsearch disagree with Postgres, Postgres wins. All derived stores are caches — buildable by replaying the Kafka event log from the beginning.
▶Why is an immutable event log a powerful architectural primitive?
Model Answer
  • Auditability. Financial systems must be able to reconstruct the exact state of any account at any historical point. With a mutable DB you only see current state — how you got there is gone. With an immutable event log you replay all events up to any timestamp and reconstruct exact historical state. This is a regulatory requirement, not just a nice-to-have.
  • Rebuilding derived views is safe and cheap. If your Elasticsearch index gets corrupted, or you add a new data store, replay the full event log through the consuming service. No migration scripts, no backfills with complex SQL. Just replay. This makes adding new consumers safe and fast.
  • Debugging production issues. "Why does account X show a balance of -£50?" With a mutable DB, the audit trail may not exist. With an event log, step through every event in sequence and identify exactly which event caused the anomaly — with timestamps, user IDs, and full context.
  • Decoupling. The payment service emits events. It doesn't know or care what consumes them. Adding a new fraud ML pipeline, a new regulatory reporter, or a new notification channel requires no changes to the payment service — just add a new Kafka consumer.
  • This is how all serious financial systems work. Double-entry accounting ledgers have been append-only for 700 years. Kleppmann's insight is that this pattern — the immutable log as source of truth — should be the default for all data systems, not just finance.
▶How do you ensure a distributed payment system doesn't double-charge users under partial failures?
Model Answer
  • Idempotency key at every layer. Client generates a UUID before initiating the payment and sends it in the request header. This key travels through the entire system. Every processing step uses it to deduplicate retries.
  • Idempotent API layer: payment service stores (idempotency_key → response) in DB before calling the PSP. On retry with the same key, return the stored response without reprocessing. The check-and-store must be atomic — use a unique DB constraint on idempotency_key with an upsert pattern.
  • Never retry blindly to PSP. A PSP call timeout means you don't know if the card was charged. Query the PSP's status endpoint with the idempotency key before retrying. If the PSP says it succeeded, return success with the existing response. If not found, retry. Retrying without checking first risks double-charging.
  • Exactly-once ledger writes. Use payment_id as a unique constraint in the ledger table. A duplicate event reaching the ledger writer fails with a constraint violation — treated as success by the consumer, not an error.
  • Reconciliation as the final safety net. A nightly batch job cross-references internal ledger entries against PSP settlement reports. Any discrepancy — a PSP charge with no ledger entry, or a ledger entry with no PSP charge — triggers an investigation workflow. This catches the bugs that slipped through all the other layers.
Client retry sequence (safe with idempotency): 1. Client sends POST /payments with Idempotency-Key: "uuid-abc" 2. Server receives, stores idempotency_key="uuid-abc" → PENDING 3. Server calls PSP → timeout (unknown outcome) 4. Client retries with same Idempotency-Key: "uuid-abc" 5. Server checks DB: key exists, status=PENDING 6. Server queries PSP for status by idempotency key 7. PSP: "charge succeeded at 14:03:22" → update to COMPLETED 8. Return success to client → Card charged exactly once. Client gets a definitive answer. → No double-charge regardless of how many retries.

▶What clarifying questions should you always ask before starting a design?

▶Why must the web tier be stateless before you can horizontally scale it?
Model Answer
  • Stateful servers require sticky sessions. If your web server holds session data in local memory, every request from a user must go to the same server — the load balancer must use sticky sessions. This breaks horizontal scaling: you can't freely add/remove servers, and if a server dies, all its sessions are lost.
  • Stateless servers are equivalent. When all session state lives in Redis, any server can handle any request. The load balancer can use round-robin or least-connections routing. Adding a server is instant — it immediately participates. Removing a server loses nothing. Autoscaling works perfectly.
  • What belongs in the session store: user authentication tokens, shopping cart state, temporary UI preferences. What doesn't: persistent data — that belongs in the DB.
Stateful (wrong): Server 1: holds session for user "alice" in memory Server 2: knows nothing about "alice" → LB must always route alice to Server 1 → Server 1 crashes → alice's session is lost Stateless (correct): Server 1, 2, 3: all identical, no local state Redis: holds {session_id → user_data} for all users → Any server handles any request → Server 1 crashes → LB routes alice to Server 2, Redis has her session ✓

Rate LimiterControlling traffic to protect downstream services

Token Bucket most common

Bucket holds max N tokens, refilled at fixed rate. Each request consumes 1. Reject if empty. Allows bursts up to bucket size. Used by AWS API Gateway, Stripe.

Other Algorithms

  • Leaky bucket: fixed output rate, queued. Smooth, no bursts
  • Fixed window: count per fixed time window. Simple but boundary burst problem
  • Sliding window log: accurate, memory-heavy
  • Sliding window counter: weighted current + previous window. Good balance

Distributed Rate Limiting the hard part

Per-node counters allow users to exceed limits by hitting multiple servers. Fix: centralised Redis with atomic INCR+EXPIRE. Or: approximate local counters synced periodically (small inaccuracy acceptable for most cases).

▶Token bucket vs sliding window — when would you choose each?
Model Answer
  • Token bucket when you want to allow controlled bursts. A user who hasn't used their quota for 30 minutes can burst up to the bucket size. Good for API rate limiting where occasional bursts are acceptable business behaviour (a user running a script that imports data).
  • Sliding window counter when you want smooth, consistent limiting with no burst allowance. Useful for protecting downstream services where even a short burst could cause issues (database connection limits, third-party API quotas). More accurate than fixed window, cheaper than sliding window log.
  • Fixed window only when simplicity is the priority and boundary burst is acceptable. Simplest to implement and explain, but a user can legitimately make 2x the allowed requests by sending 1000 at 11:59pm and 1000 at 12:00am.
▶How do you handle rate limiting in a microservices architecture where each service has different limits?

  • Per-route limits: define limits at the route level in the gateway. POST /payments = 100/min (expensive operation). GET /balance = 1000/min (cheap read). GET /transactions = 200/min (moderate).
  • Shared counter space in Redis. All gateway nodes share a Redis cluster for counter storage. Key structure: rate:{user_id}:{service}:{endpoint}:{window}. Allows independent limits per service per user.
  • Service-level limits (defence in depth): for critical services (payments, auth), also implement rate limiting at the service level as a second layer. The gateway handles normal traffic; the service-level limit protects against gateway bypasses or internal service-to-service calls.

Consistent HashingDistributing data across nodes with minimal reshuffling

── Hash ring (0 → 2³²) ────────────────────────────── 0 ↗ ↘ Server A Server B ← servers placed by hash(server_ip) ↑ ↓ Server D Server C ↖ ↙ 2³² Key X → hash(X) → walk clockwise → first server = home server Add node E: only keys between E's predecessor and E move → minimal reshuffling

Interview Questions & Model Answers — click to expand
▶Why does consistent hashing minimise data movement when nodes are added or removed?
Model Answer
  • With hash mod N: adding one node changes N, so hash(key) mod N changes for almost every key — nearly every piece of data has to move to a new node. Adding a node to a 10-node cluster means reshuffling ~91% of all data.
  • With consistent hashing: keys and nodes are placed on a ring using the same hash function. A key is assigned to the first node clockwise from its ring position. When you add a new node, only the keys between the new node and its predecessor need to move — that's 1/N of all keys on average. Removing a node moves only that node's keys to its successor.
  • Why 1/N is significant: for a 100-node cache cluster, adding a node moves ~1% of cached data. With hash mod N it would be ~99%. The difference is a cache hit rate drop from 99% to nearly 0% during rebalancing — catastrophic for a hot cache.
10-node cluster, adding node 11: Hash mod N: ~91% of keys get a new home → must be fetched from DB → cache miss storm Consistent: ~10% of keys get a new home → 90% cache hit rate maintained during rebalance This is why Redis Cluster, Cassandra, and DynamoDB all use consistent hashing variants.
▶What problem do virtual nodes solve?
Model Answer
  • The uneven distribution problem. When you place N physical nodes on a ring by hashing their IDs, they land randomly and unevenly. Some nodes end up with large arc ranges (responsible for many keys); others with tiny ranges. This means uneven load distribution — some nodes handle 3x the traffic of others.
  • Virtual nodes (vnodes): each physical node gets 100–200 positions on the ring instead of 1. The positions are distributed evenly (or weighted by node capacity). Each arc between positions is small. Load is spread across many small chunks rather than a few large ones.
  • Second benefit — smooth rebalancing: when you add a physical node, it takes over many small vnodes from many different existing nodes. The load is absorbed from the entire cluster, not just one neighbour. Much smoother redistribution.
  • Third benefit — heterogeneous nodes: a node with 2x the RAM can be assigned 2x the vnodes, giving it 2x the load — a simple way to weight nodes by capacity without any extra logic.
▶Where would you use consistent hashing vs simple hash-mod-N sharding?
Model Answer
  • Use consistent hashing when the number of nodes changes dynamically — autoscaling, nodes failing and recovering, planned scaling events. Cache clusters and distributed KV stores are the classic cases. You need rebalancing to be cheap and non-disruptive.
  • Hash mod N is fine when the number of nodes is fixed and rarely changes, or when you're willing to accept a full reshard when you add nodes (e.g. a small sharded Postgres cluster where resharding is a planned maintenance window once a year). Simpler to implement and reason about.
  • Fixed partition mapping is often the best compromise for databases: create 1000 logical partitions upfront, assign them to physical nodes. When you add a node, reassign some logical partitions to it. This decouples rebalancing from rehashing.

▶How does a gossip protocol detect node failures without a central coordinator?
Model Answer
  • Each node maintains a membership list with a heartbeat counter for every other node. Every T seconds, each node picks a few random peers and exchanges its membership list. Peers merge the lists, taking the maximum heartbeat count per node.
  • Failure detection: if a node's heartbeat counter hasn't increased in X seconds (across multiple gossip rounds), its neighbours mark it as "suspected dead." After a longer timeout, it's marked "dead" and removed from routing tables.
  • Why gossip over central coordinator: a central coordinator is a single point of failure. Gossip is decentralised — there's no single component whose failure breaks failure detection. It's also O(log N) messages per round to propagate state to the whole cluster.
  • Real systems: Cassandra uses gossip for cluster membership. Each node knows about every other node. A node joining sends a gossip message that propagates to the whole cluster within seconds without any central registry.
▶Explain vector clocks. How do they help detect conflicting writes?
Model Answer
  • Vector clock: a map from node ID to logical counter, attached to every write. When a node writes, it increments its own counter. When it receives a message, it takes the element-wise maximum of its clock and the incoming clock.
  • Detecting causality: clock A "happened before" clock B if every entry in A is ≤ the corresponding entry in B. This means B causally follows A — safe to discard A.
  • Detecting conflict: if neither clock dominates the other (some entries of A are higher, some of B are higher), the writes are concurrent — there's a genuine conflict that requires resolution.
  • Application-level resolution: on conflict, return both versions to the client. The client (or application logic) merges them. Amazon's shopping cart used this: two concurrent "add item" events are both shown and the user sees items from both versions merged.
Node A writes value: clock=[A:1, B:0, C:0] Node B also writes concurrently: clock=[A:0, B:1, C:0] Compare: A:1>0 in version 1, B:0<1 in version 1 → neither dominates → Concurrent writes → CONFLICT → return both to client for resolution Node A then reads and merges: clock=[A:2, B:1, C:0] → This new write causally follows both → conflict resolved

▶Design a unique ID generator for sortable IDs across 20 datacenters.
4-Step Framework Answer

  2. With 20 datacenters: 5 bits = 32 possible DCs, so 20 fits. Assign each DC a unique 5-bit ID at deployment time. Workers within each DC get unique 5-bit IDs.
  3. Sequence number: resets to 0 each millisecond. If 4,096 IDs/ms/worker isn't enough, expand the sequence field at the cost of shrinking another field.

Step 4 — Wrap Up: Clock Skew Risk
  1. Clock skew is the main risk — if a worker's clock goes backward, sequence numbers could collide with previously-issued IDs
  2. Mitigation: if current_time < last_issued_time, refuse to generate IDs until the clock catches up. Log an alert. This creates a brief pause but prevents duplicates.
▶What happens to Snowflake IDs if the system clock skews backwards?
Model Answer
  • Clock skew backward = duplicate ID risk. If a worker issued ID with timestamp=T=1000 and the clock is then corrected to T=990, the next ID generated uses T=990 — which is earlier than the last issued timestamp. With the same sequence counter starting at 0, you could generate the same (timestamp, worker_id, sequence) combination as a previously-issued ID.
  • Mitigation 1 — Refuse and wait: the generator tracks the last issued timestamp. If current_time < last_timestamp, refuse to issue IDs and sleep until current_time >= last_timestamp. A 10ms clock skew causes a 10ms pause — acceptable for most systems.
  • Mitigation 2 — Logical clock: always increment the "timestamp" portion, never go backward. If the physical clock goes back, use last_timestamp + 1 until the physical clock catches up. IDs remain unique but are no longer strictly time-ordered.
  • Mitigation 3 — Disable NTP stepping, use slewing: configure NTP to use "slewing" (gradual clock adjustment) rather than "stepping" (instant jump). The clock corrects slowly over minutes, never jumping backward suddenly.
▶Why not just use UUIDs? What properties do you lose?
Model Answer
  • Not time-sortable. UUID v4 is purely random. You can't sort records by creation time using their ID — you need a separate created_at column and index. Snowflake IDs are naturally sorted by generation time, making range queries and pagination by ID trivially efficient.
  • Index fragmentation. Random UUIDs cause significant B-tree index fragmentation in relational databases. New UUIDs are inserted at random positions in the index, causing frequent page splits and poor cache locality. Sequential IDs (Snowflake, ULID) insert at the end — no fragmentation.
  • Size. UUID is 128 bits (16 bytes) vs Snowflake's 64 bits (8 bytes). In a table with billions of rows and multiple UUID foreign keys, this doubles index size and memory footprint. Not trivial at scale.
  • When UUIDs are fine: small-to-medium scale systems where simplicity beats performance, or when IDs are exposed publicly and you want to avoid leaking creation time or rate information (UUIDs reveal nothing; Snowflake timestamps reveal when an entity was created).

▶How do you deduplicate 10 billion URLs efficiently without storing them all?
Model Answer
  • Bloom filter. A probabilistic data structure that uses k hash functions over a bit array. To check membership: hash the URL k times, check all k positions in the bit array. If any are 0, the URL has definitely not been seen. If all are 1, it's probably been seen (small false-positive rate, no false negatives).
  • Memory: for 10B URLs with 1% false-positive rate, a Bloom filter needs ~12GB of RAM. Storing the URLs themselves (at 100 bytes each) would need 1TB. A ~80x memory reduction.
  • Content deduplication with SimHash: the same content at two different URLs (mirrors, syndicated articles) should be crawled but not re-indexed. SimHash produces a 64-bit fingerprint that is similar for near-duplicate documents (changing a few words changes few bits). Two pages with <3 bit difference are considered duplicates.
  • DDIA connection: Bloom filters are used in LSM-tree storage engines (Cassandra, RocksDB) for exactly the same reason — to avoid expensive disk reads for keys that probably don't exist in an SSTable.

News Feed (Facebook / Instagram)Fan-out on write vs fan-out on read — the central trade-off

Fan-out on write (push) fast reads

When user A posts, pre-compute and push to all followers' feed caches. Reads are O(1) — just read the cache. Write cost: O(followers). Breaks for celebrities (50M followers = 50M cache writes per post).

Fan-out on read (pull) fast writes

When user reads feed, pull recent posts from all followed users and merge. Write is cheap. Read is O(following) — N DB queries, slow for users following thousands.

Hybrid (what Facebook/Instagram actually do) correct answer

Fan-out on write for normal users. Fan-out on read for celebrities (>threshold followers). At read time, merge the pre-computed feed cache with fresh celebrity posts.

▶Why use WebSockets over HTTP long-polling for a chat system?
Model Answer
  • HTTP long-polling (the alternative): client sends a request, server holds it open until there's a message to deliver, then responds. Client immediately sends another request. Simulates real-time but uses HTTP — works through any proxy, simple to implement. Downside: each "held" request consumes a thread/connection on the server, and there's request overhead on every message delivery.
  • WebSockets: a persistent, full-duplex TCP connection established via an HTTP upgrade. After the handshake, both client and server can push messages at any time with minimal overhead (small frame header, no HTTP headers per message). One persistent connection per client vs one HTTP request per message delivery.
  • Why WebSockets win for chat: (1) server-initiated push is native — the server pushes messages to the client the instant they arrive, with no polling delay; (2) much lower per-message overhead (no HTTP headers); (3) better for high-frequency bidirectional communication (typing indicators, read receipts, presence updates).
  • Downside of WebSockets: stateful — the client is pinned to one chat server for the lifetime of the connection. Server autoscaling requires careful connection migration. Load balancers must support sticky sessions or WebSocket-aware routing.

Proximity Service (Yelp / Find Nearby)Geospatial indexing — Geohash vs Quadtree

Geohash most common approach

Encode lat/lng as a short string by recursively dividing the world into a grid. Nearby places share a common prefix.

# Precision levels Length 4: ±20km (city-level) Length 6: ±0.6km (neighbourhood) Length 8: ±19m (building-level)

Edge case: two nearby places can share no common prefix if they straddle a grid boundary. Fix: search current cell + 8 surrounding cells.

Recursively subdivide space into 4 quadrants until each contains ≤ N businesses. Adapts to density — cities get finer cells, oceans get coarse ones. In-memory tree structure. Used for dynamic/moving objects.

▶Compare Geohash vs Quadtree — when would you choose each?
Model Answer
  • Geohash strengths: simple string representation makes it easy to store in any DB or cache. Prefix queries work natively in any key-value store (SCAN geohash:gcpv*). Redis has built-in geospatial support based on Geohash. Good for static data (restaurants, businesses) that doesn't move frequently.
  • Geohash weakness — boundary problem: two adjacent cells can have completely different prefixes if they straddle a grid boundary. You must always search 9 cells (current + 8 neighbours) to avoid missing nearby results near boundaries. Extra complexity.
  • Quadtree strengths: adapts to data density — high-density urban areas get finer subdivision, sparse rural areas get coarse cells. No fixed grid size. Better for non-uniform data distributions. Good for dynamic data (moving objects, live locations) because the tree can be updated incrementally.
  • Quadtree weakness: in-memory tree structure is harder to distribute across nodes. Requires careful serialisation for persistence. More complex to implement and maintain than Geohash.
  • Practical choice: Geohash + Redis for most proximity search use cases (businesses, POIs). Quadtree for ride-sharing with moving vehicles or game world object lookup where density varies enormously.
▶What's the boundary problem in Geohash and how do you fix it?
Model Answer
  • The problem: Geohash divides the world into a grid. Two places that are 10 metres apart but on opposite sides of a grid boundary will have completely different Geohash prefixes. A prefix-only search for one would miss the other entirely.
  • Example: a restaurant at Geohash gcpvh and a coffee shop at Geohash gcpvj are adjacent cells — 200 metres apart. But their prefixes share only 4 characters. If you search prefix gcpvh, you find the restaurant but not the coffee shop.
  • Fix — always search 9 cells: for any user location, compute the Geohash of their position, then compute the 8 neighbouring cell hashes. Search all 9 cells and merge the results. Every library that works with Geohash has a neighbours() function. This guarantees you never miss a nearby result due to a boundary split.
  • Why 9 and not more: at precision 6 (±600m cells), searching 9 cells covers a 3×3 grid — roughly 1.8km × 1.8km area. For a 500m radius search, this always includes all results within the radius regardless of where the user sits within their cell.

  • A*: heuristic (straight-line to destination) makes it faster

Interview Questions & Model Answers — click to expand
▶How does Google Maps calculate the fastest route across an entire country in under a second?
Model Answer
  • Naïve Dijkstra is too slow. A road network for a country has ~10–100M nodes. Full Dijkstra explores the entire graph — would take minutes. Even A* with a straight-line heuristic is too slow for cross-country routes.
  • Contraction Hierarchies (CH): pre-process the graph offline by ranking nodes by "importance" (how many shortest paths pass through them). Motorway junctions are very important; residential dead-end streets are unimportant. Build a hierarchy by "contracting" unimportant nodes — replacing them with shortcut edges between their neighbours.
  • Bidirectional search: run A* forward from the source and backward from the destination simultaneously. They meet in the middle. On a hierarchical graph, this converges very quickly — the search only needs to explore high-importance nodes (motorways, major roads) for most of the route.
  • Result: routes that would take Dijkstra millions of node explorations are solved in thousands — sub-100ms for cross-country routes on modern hardware. This offline preprocessing is why Google can route across continents instantly.
  • Traffic updates: CH pre-computation uses historical average edge weights. Real-time traffic updates adjust edge weights dynamically. For major traffic changes, a lightweight re-query on the affected subgraph updates the route without full recomputation.
▶How do you incorporate real-time traffic into routing without recomputing from scratch?
Model Answer
  • GPS probe data pipeline. Every Google Maps user's phone (with location sharing enabled) sends anonymised speed data every few seconds. This is the real-time traffic source — billions of data points/day. A Kafka + Flink pipeline aggregates speed per road segment every 30–60 seconds.
  • Edge weight updates. The routing graph's edge weights (travel time) are updated in near-real-time from the aggregated probe data. A road segment where users are averaging 5mph instead of 40mph gets its edge weight updated accordingly.
  • Selective recomputation. You don't rebuild the entire CH graph. Instead, maintain a "live traffic" overlay on top of the historical CH graph. Routing queries use historical CH for most of the graph and apply live traffic adjustments for segments near the start and end of the journey (where traffic conditions are most relevant).
  • ETA refinement during navigation: as the user drives, ETA is recalculated every few minutes using their current position, updated traffic, and remaining route. If a major jam appears ahead, an alternative route is proposed.
▶How would you handle map tile serving for 1B mobile users?
Model Answer
  • CDN is the entire answer for hot tiles. Popular tiles (major cities, zoom levels 10–15) are cached at CDN edge nodes globally. A cache hit at the CDN edge never reaches your origin. With billions of users, the most popular tiles have near-infinite cache hit rates — the same London city view is requested millions of times.
  • S3 for origin storage. Tiles pre-rendered and stored in S3 at all zoom levels. S3 key structure: /tiles/{z}/{x}/{y}.png — directly maps to the tile coordinate system. CDN pulls from S3 on cache miss.
  • Vector tiles for mobile. Sending raw geometry (roads, buildings, labels as vector data) instead of pre-rendered PNGs. The client renders on-device using WebGL/Metal. Benefits: smaller payload (geometry compresses better than images), resolution-independent (no separate tiles for Retina displays), and the map can be styled on-device without server changes.

▶Design a distributed message queue. What are the core trade-offs between Kafka and RabbitMQ?
4-Step Framework Answer
Step 1 — Clarify
  1. Push vs pull consumption?
  2. Do consumers need to replay messages?
  3. Ordering guarantees needed?
  4. Expected message size and throughput?
  5. At-least-once or exactly-once?
Step 2–3 — Kafka design decisions
  1. Append-only log on disk: sequential writes are extremely fast even on spinning disk. Immutable log means no locking for concurrent reads. Messages retained for configurable period (days/weeks) — enables replay.
  2. Pull-based consumption: consumers control their own pace. Can batch reads for efficiency. No back-pressure problem (a slow consumer doesn't overwhelm the broker). Consumers simply lag behind and catch up when capacity allows.
  3. Consumer groups: each partition consumed by exactly one consumer per group. Different groups read independently with their own offsets. Adding consumers scales throughput; adding groups scales fan-out.
  4. Durability: replication factor 3. Producer waits for ISR (in-sync replicas) acks before confirming. Leader + 2 replicas means 2 node failures before data loss.

▶How does Kafka achieve exactly-once delivery? What components are involved?
Model Answer
  • Three components work together: idempotent producer, transactional producer API, and read_committed consumer isolation.
  • Idempotent producer: each producer session gets a producer ID (PID). Each message gets a sequence number. The broker tracks (PID, partition, sequence) and deduplicates retries. If a producer retries after a timeout, the broker recognises the sequence number and returns the original ack without reprocessing. Eliminates duplicates from producer retries.
  • Transactional producer: wraps multiple produce operations (to multiple topics/partitions) + consumer offset commits in an atomic transaction. Either all succeed or all are rolled back. Uses a two-phase commit protocol internally between the producer and transaction coordinator (a special broker). The transaction coordinator's log is itself a Kafka topic — durable.
  • Read_committed consumer: consumers with isolation.level=read_committed only see messages from committed transactions. Messages from aborted transactions are never visible. This prevents consumers from processing messages that were later rolled back.
  • End-to-end exactly-once: this covers Kafka-to-Kafka pipelines (consume, transform, produce). For Kafka-to-external-DB pipelines, you still need idempotency at the DB write — Kafka transactions alone don't cover the last mile.
▶A consumer is processing too slowly and falling behind. How do you diagnose and fix consumer lag?
Model Answer
  • Consumer lag = (latest offset) - (consumer committed offset) per partition. Monitor with kafka-consumer-groups.sh or Kafka's built-in metrics. Alert when lag exceeds a threshold (e.g. more than 10 minutes of messages unprocessed).
  • Diagnose the root cause first:
    • Is the consumer CPU-bound? (Expensive processing per message)
    • Is it I/O-bound? (Slow DB write, slow downstream HTTP call)
    • Is it a single "poison pill" message causing retries that block the partition?
    • Did a single slow consumer get assigned too many partitions?
  • Fix 1 — Add consumer instances: if the topic has more partitions than consumers, add consumers to the group. Each gets a subset of partitions. But: you can't have more consumers than partitions — excess consumers sit idle.
  • Fix 2 — Add partitions: increase partition count to allow more parallelism. Warning: this redistributes partition assignments and can cause a brief rebalance pause. Also: message ordering is per-partition, so adding partitions may change the ordering guarantees.
  • Fix 3 — Async processing: if the bottleneck is a slow downstream call (DB write, API call), use async processing within the consumer — batch messages and write them in parallel using thread pools, then commit the batch offset once all writes succeed.
  • Fix 4 — Dead letter queue for poison pills: if one malformed message keeps failing and blocking a partition, route it to a dead letter topic after N retries. Continue processing the rest of the partition. Investigate the DLQ separately.
▶Why does Kafka use pull-based consumption rather than push?
Model Answer
  • Consumer controls its pace. With push, the broker pushes messages at whatever rate they arrive. A slow consumer gets overwhelmed — either it drops messages or applies back-pressure to the entire pipeline. With pull, the consumer requests the next batch only when it's ready. A slow consumer just processes slower and accumulates lag — it doesn't affect other consumers or the broker.
  • Efficient batching. With pull, the consumer can request a batch of N messages in one round trip. The broker waits (up to a configurable max wait time) for enough messages to fill a batch. This amortises network round-trip cost across many messages — much more efficient than push, which must push each message as it arrives or buffer them.
  • Simplifies the broker. The broker doesn't need to track the state of each consumer connection or implement back-pressure protocols. It just serves messages on request. State (current offset) is owned by the consumer.
  • Replay is natural. A consumer that wants to reprocess from offset 0 just starts requesting from offset 0. No broker-side configuration change needed — the consumer just changes its offset. With push, the broker would need to know where to start pushing.

▶What is high cardinality and why is it a problem for metrics systems?
Model Answer
  • Cardinality = number of unique label value combinations. A metric with labels {service, endpoint, status_code} might have 10 services × 50 endpoints × 10 status codes = 5,000 unique time series. That's low cardinality — fine.
  • High cardinality example: adding user_id as a label to a request counter. With 10M users, you have 10M unique time series for that one metric. Each time series needs its own storage, indexing entry, and memory allocation in the TSDB. Prometheus can handle ~10M time series before performance degrades severely.
  • Why it kills performance: every unique label combination is a separate time series — a separate row in the TSDB index. Querying all time series for a metric (e.g. "sum of requests across all users") requires reading millions of series and aggregating — extremely expensive. The index itself becomes huge.
  • Fix: don't put high-cardinality values in metric labels. Instead: use a separate distributed tracing system (Jaeger, Zipkin) for per-request/per-user data. Keep metrics for aggregate measurements; use traces for individual request analysis.

▶What is erasure coding and why is it better than 3x replication for large files?
Model Answer
  • 3x replication: store 3 full copies of every file. Lose any 2 copies and you still have 1. Storage overhead: 3x. Simple to implement and recover — just copy from a surviving replica.
  • Erasure coding (Reed-Solomon): split data into k data shards and m parity shards. Any k of the k+m shards can reconstruct the original data. Common configuration: 8+4 (8 data, 4 parity). Store 12 shards instead of 3 copies. Storage overhead: 12/8 = 1.5x instead of 3x.
  • Durability comparison (8+4 vs 3x replication): both tolerate losing 4 nodes. But erasure coding uses half the storage. At petabyte scale, this difference is enormous — a 50PB storage system needs 75PB with erasure coding vs 150PB with 3x replication.
  • Downside of erasure coding: reconstruction on read (when shards are missing) requires fetching k shards and computing the parity — more CPU than just reading a replica. Also, more complex to implement correctly. This is why S3 and Google Cloud Storage use erasure coding for cost efficiency at scale, while smaller systems often use replication for simplicity.
▶How does S3 achieve 11 nines of durability?
Model Answer
  • 11 nines = 99.999999999% durability means losing one object out of 100 billion per year. In practice, S3 has never had a confirmed data loss event for properly stored objects.
  • Multiple AZs: data is replicated across at least 3 Availability Zones within a region. Each AZ has independent power, cooling, and networking. A single AZ failure doesn't affect data — the other 2 AZs have complete copies.
  • Erasure coding within each AZ: within each AZ, data is further erasure coded across multiple storage nodes. Even if multiple disks fail simultaneously within an AZ, data is recoverable.
  • Continuous data integrity checks: S3 runs background scrubbing — regularly reads and re-verifies checksums of all stored objects. Bit rot (silent data corruption from cosmic rays or disk degradation) is detected and repaired by reading from a good replica/shard.
  • Redundant disk controllers and network paths: no single hardware component failure can cause data loss. Every storage path has redundancy at every layer.
  • What 11 nines doesn't cover: user error (accidentally deleting objects). S3 Versioning and Object Lock are the features that protect against this — 11 nines only covers hardware/software failure durability.

▶A payment request times out. The client retries. How do you prevent double-charging?

▶What is a double-entry ledger and why do payment systems use it?
Model Answer
  • Double-entry accounting: every financial transaction creates exactly two ledger entries — a debit (money leaves an account) and a credit (money enters another account). The sum of all debits always equals the sum of all credits — the books always balance.
  • Why payment systems use it: (1) any discrepancy is immediately detectable — if debits and credits don't balance, money has been created or destroyed by a bug; (2) every balance can be computed by summing ledger entries for an account — no need to trust a stored "balance" field that could get out of sync; (3) the ledger is append-only — you never update or delete entries, only add new ones. Corrections are themselves new entries.
  • Practical implementation: a ledger table with columns: (entry_id, transaction_id, account_id, amount, direction, timestamp, description). A payment of £100 from alice to bob creates two rows: (tx_123, alice, -100, DEBIT) and (tx_123, bob, +100, CREDIT). The transaction_id links them.
  • Balance computation: SELECT SUM(amount) FROM ledger WHERE account_id = 'alice'. Negative = debit balance, positive = credit. This is always correct because it's derived from the immutable ledger history.
Ledger entries for a £100 payment from Alice to Bob: entry_id | tx_id | account_id | amount | direction | timestamp 1001 | tx_123 | alice | -100.00 | DEBIT | 2024-06-18 14:05:00 1002 | tx_123 | bob | +100.00 | CREDIT | 2024-06-18 14:05:00 CHECK: SUM of all amounts for tx_123 = -100 + 100 = 0 ✓ (money conserved) Alice balance = SUM(amount) WHERE account='alice' = previous + (-100) Bob balance = SUM(amount) WHERE account='bob' = previous + (+100)

    • Ingest PSP settlement file (CSV/SFTP) → parse → load into reconciliation DB table
    • Query internal ledger for same date range

▶Why can't you read balance from a replica before authorising a spend?
Model Answer
  • Replication lag means the replica may be stale. If alice has £100 and spends £80, the leader commits the debit immediately. The replica may still show £100 for the next few milliseconds (or seconds under heavy load). If you read the balance from the replica, it shows £100 — alice appears to have enough for a second £80 spend. You authorise it. Now alice is -£60.
  • This is the read-your-own-writes problem at the worst possible moment. DDIA Ch 5 describes this exactly. For financial balances, you need to read from the leader (or use a linearizable read) to guarantee you see the latest committed balance.
  • The check-then-act must be atomic. Even reading from the leader isn't enough if the check and the debit are in separate transactions. Between checking the balance and debiting it, another request could debit it. The check and debit must be in the same transaction: UPDATE wallets SET balance = balance - 100 WHERE user_id='alice' AND balance >= 100. If 0 rows affected, insufficient funds. No separate SELECT needed.
  • Acceptable places to use replica reads: displaying balance history, generating statements, analytics queries. Any read that's used to make an irreversible financial decision must go to the leader.

  • Design principle: compensating transactions must be simpler and more reliable than the original transaction. The original credit might involve fraud checks, FX conversion, compliance checks. The compensation (refund) should bypass all of these and just write the credit directly to minimise failure surface.

  3. Match condition: top bid price ≥ lowest ask price → trade executes at ask price, FIFO within price level.

  2. No GC: Java GC pauses are fatal for microsecond latency. Either use C++, or Java with off-heap memory and GC tuning (G1GC, very large heap to minimise GC frequency).
  3. Kernel bypass: use DPDK (Data Plane Development Kit) to bypass the Linux network stack. Reduces network latency from ~20µs to ~2µs.

▶Why must the matching engine be single-threaded?
Model Answer
  • Correctness requires total ordering of operations. The matching engine must apply orders in a deterministic sequence. If two orders arrive simultaneously and both could match the same resting order, exactly one must win. With multiple threads, this requires locking on the order book — and lock contention introduces unpredictable latency spikes.
  • Locking is incompatible with microsecond targets. A mutex lock in the critical path adds ~50–200ns overhead per acquisition on a busy system. At 1M orders/sec, that's 50–200ms of added latency per second of throughput — unacceptable. Lock-free data structures help but add complexity.
  • Single-threaded is actually fast enough. A modern CPU can process millions of simple operations per second. The matching engine is primarily memory operations — array lookups, linked list pointer updates. 1M orders/sec in a tight single-threaded loop is achievable in C++ (LMAX achieved millions/sec on commodity hardware with their Disruptor pattern).
  • Multiple instances for multiple assets: you can have one matching engine thread per asset (AAPL, GOOG, TSLA each get their own single-threaded engine). Orders for different assets never interact, so parallelism is achieved across assets without any locking.
▶How do you recover the order book state after a matching engine crash?
Model Answer
  • Event sourcing. The order book's state is entirely determined by the sequence of order events it has processed. Keep an immutable, append-only log of all order events (new order, cancel, partial fill, full fill). The log is written to durable storage before the matching engine processes each event.
  • Snapshots to speed up recovery. Replaying the full order log from the beginning of time would take too long. Periodically (e.g. every 5 minutes) serialize the current order book state to a snapshot file (S3 or fast local storage). On crash, load the latest snapshot (restoring state as of 5 minutes ago), then replay only the events since the snapshot.
  • Sequencer as the source of truth. All order events are first written to a sequencer (assigns a monotonically increasing sequence number to each event). The matching engine only processes events in sequence order. On recovery, the engine fetches events starting from its last processed sequence number — the sequencer ensures it doesn't miss or duplicate any.
  • Hot standby. A passive replica matching engine receives the same event stream from the sequencer and applies it to its own order book state. On primary failure, the standby becomes active from its current state — failover in milliseconds without needing to replay from a snapshot.
Recovery process after matching engine crash: 1. Load latest order book snapshot (e.g. 14:25:00 state) from S3 2. Fetch all sequencer events from sequence_id=N onwards (where N was the last event in the snapshot) 3. Replay events: sequence 1001 (new order AAPL BUY 100@$150), 1002 (cancel order 501), ... 4. Order book is reconstructed to exact state at crash time 5. Resume processing from current sequence_id With 5-minute snapshots and 1M events/sec, worst-case replay = 5min × 1M = 300M events At 10M events/sec replay speed → 30 seconds to recover. Acceptable.
▶Two orders arrive simultaneously at microsecond resolution — how do you determine priority?
Model Answer
  • The sequencer is the arbiter. All orders pass through a sequencer before reaching the matching engine. The sequencer assigns a monotonically increasing sequence number to each order as it arrives. Ties in arrival time are broken by the sequence number — whichever order gets the lower sequence number was "first."
  • Price-time priority (FIFO within price level): orders at the same price compete by their sequence number. The order with the lower sequence number (arrived earlier at the sequencer) gets priority. This is the standard FIFO price-time priority used by most exchanges.
  • What "simultaneous" means at microsecond resolution: true simultaneity is impossible — even at nanosecond resolution, the sequencer processes one message at a time (single-threaded). "Simultaneous" means both arrived in the same millisecond from the client's perspective, but the sequencer's monotonic counter determines the strict order.
  • Co-location advantage: trading firms pay to physically co-locate their servers in the exchange's data centre. A server 100m away from the sequencer gets messages there ~350ns sooner than one 1km away (speed of light). This microsecond advantage in sequence number assignment is the literal product that co-location vendors sell.
