The Interview FrameworkThe 4-step process Alex Xu uses for every single design question

The 4-Step Framework — use this structure every time

  1. Understand the problem & establish scope (3–5 min) — Ask clarifying questions. Don't jump straight to solutions. What features matter? What scale? What are the edge cases? Read/write ratio? Mobile vs web?
  2. Propose high-level design & get buy-in (10–15 min) — Draw a simple box diagram. Identify the core components. Talk through the happy path. Get the interviewer to agree before diving deep.
  3. Design deep dive (15–20 min) — Pick 2–3 components to go deep on. Let the interviewer guide if possible. Cover data model, API design, concurrency, failure modes.
  4. Wrap up (3–5 min) — Identify bottlenecks, talk about monitoring, discuss future improvements. Show you think operationally.
⚠️ What kills candidates: jumping to solutions before asking clarifying questions, over-engineering from the start, not knowing back-of-envelope numbers, and going quiet instead of thinking out loud.

Clarifying questions to always ask

  • How many daily active users (DAU)?
  • What's the read-to-write ratio?
  • Is this a global system or single region?
  • What's the latency requirement (p99)?
  • Do we need strong consistency or is eventual OK?
  • What's the data retention period?
  • Mobile, web, or both?

Standard components to know cold

  • DNS + CDN — edge caching, GeoDNS routing
  • Load balancer — L4 vs L7, sticky sessions, health checks
  • API Gateway — auth, rate limiting, routing
  • Cache — Redis/Memcached, eviction policies, cache-aside vs write-through
  • Message queue — Kafka, RabbitMQ, async decoupling
  • Object storage — S3 for blobs, not for structured data
  • Search — Elasticsearch for full-text, not your primary DB

Scale from Zero to MillionsThe canonical evolution of a web system as traffic grows

── Stage 1: Single server ────────────────────────────────── User → DNS → Web Server + DB (same box) ── Stage 2: Separate DB ──────────────────────────────────── User → DNS → Web Server → DB (MySQL) ── Stage 3: Load balancer + horizontal web tier ──────────── User → DNS → Load Balancer → [Web 1] [Web 2] → Primary DB ↓ replication [Replica DB] ← reads ── Stage 4: Add cache + CDN ──────────────────────────────── Static assets → CDN Hot data → Redis Cache Reads → Cache first, miss → DB ── Stage 5: Multiple data centres ────────────────────────── GeoDNS routes user to nearest DC Each DC has its own web/cache/DB tier
◈ DDIA link — Replication (Ch 5): The primary/replica DB split here is exactly single-leader replication. Reads from replicas introduce lag. The cache-aside pattern is how you mitigate read-your-own-writes problems without hitting the leader on every read.

Stateless web tier key principle

Web servers must be stateless — any server can handle any request. Session state lives in Redis, not in-process memory. This lets you add/remove web servers freely behind the load balancer.

Database scaling options

  • Vertical scaling: bigger instance. Simple but has a ceiling and is a single point of failure
  • Read replicas: offload read traffic. Works well for read-heavy workloads
  • Sharding: horizontal split by user_id or similar. Necessary at extreme scale
  • CQRS: separate read and write models entirely

Rate LimiterControlling traffic to protect downstream services

Token Bucket most common

Bucket holds max N tokens. Tokens added at fixed rate. Each request consumes 1 token. If empty, reject. Allows bursts up to bucket size.

Used by: AWS API Gateway, Stripe API rate limiting

# Redis implementation tokens = GET user:{id}:tokens if tokens > 0: DECR user:{id}:tokens allow request else: reject with 429

Other Algorithms

  • Leaky bucket: requests queued at fixed output rate. Smooth output, no bursts. Used in network QoS
  • Fixed window counter: count per fixed time window. Simple but burst at window boundaries
  • Sliding window log: log of timestamps, count in window. Accurate but memory-heavy
  • Sliding window counter: weighted average of current + previous window count. Good balance

Distributed Rate Limiting the hard part

With multiple API gateway nodes, a per-node counter is wrong — a user could hit all N servers and bypass the limit. Solutions:

  • Centralised store (Redis): all nodes read/write to one Redis. Atomic INCR + EXPIRE. Simple but Redis is now a single point of failure and adds latency to every request
  • Redis Cluster: shard counters across Redis nodes. Higher availability
  • Approximate local + sync: each node tracks locally, syncs to central store every N seconds. Small inaccuracy is acceptable for most use cases

Interview questions

  • Design a rate limiter that limits 1000 requests/minute per user across 10 API servers.
  • Token bucket vs sliding window — when would you choose each?
  • How do you handle rate limiting in a microservices architecture where each service has different limits?

Consistent HashingDistributing data across nodes with minimal reshuffling

◈ DDIA link — Partitioning (Ch 6): This is the core of DDIA's consistent hashing section. Alex Xu provides the visual explanation; DDIA provides the trade-off analysis. Together they cover the full picture.

How it works

── Hash ring (0 to 2³²) ────────────────────────── 0 ↗ ↘ Server A Server B ← servers placed by hash(server_ip) ↑ ↓ Server D Server C ↖ ↙ 2³² Key X → hash(X) = position → walk clockwise → first server = home server Adding Server E: only keys between predecessor(E) and E move → minimal reshuffling Removing Server A: only keys owned by A move to next server clockwise

Virtual nodes: each physical server gets 100–200 positions on the ring. This smooths out the distribution and makes rebalancing more even when servers have different capacities. Used by Cassandra and DynamoDB.

Interview questions

  • Why does consistent hashing minimise data movement when nodes are added or removed?
  • What problem do virtual nodes solve?
  • Where would you use consistent hashing vs simple hash-mod-N sharding?

Key-Value Store (Dynamo-style)Design a distributed KV store like Redis Cluster or DynamoDB

◈ DDIA links — Ch 5 (Leaderless replication), Ch 6 (Partitioning), Ch 9 (CAP/linearizability): This chapter is essentially a worked example of DynamoDB's original design. All the DDIA theory — quorums, sloppy quorums, consistent hashing, vector clocks, gossip protocol — shows up here in concrete form.

Core design decisions

  • Partitioning: consistent hashing across nodes
  • Replication: leaderless, any node can accept writes
  • Consistency: tunable (r + w > n = strong; r=1,w=1 = fast but eventual)
  • Conflict resolution: vector clocks track causality; last-write-wins as fallback
  • Failure detection: gossip protocol — each node pings random peers, propagates membership state
  • Failure handling: sloppy quorum + hinted handoff

Storage engine inside a node

  • Memory cache: hot keys served from in-memory hash table
  • WAL: write-ahead log for crash recovery
  • SSTable + LSM-tree: persistent storage (same as Cassandra, RocksDB)
  • Bloom filter: per-SSTable. Quickly answer "does key X exist here?" — avoids expensive disk reads for missing keys
  • Compaction: background merge of SSTables

Interview questions

  • Design a distributed key-value store that supports get/put with configurable consistency.
  • How does a gossip protocol detect node failures without a central coordinator?
  • Explain vector clocks. How do they help detect conflicting writes?

Snowflake ID structure know this cold

64 bits total: [1 bit sign] [41 bits timestamp ms] [5 bits DC] [5 bits worker] [12 bits sequence] → 41 bits = 2^41 ms ≈ 69 years from epoch → 5+5 bits = 32 DCs × 32 workers = 1,024 generators → 12 bits = 4,096 IDs per ms per generator = ~4M IDs/sec per worker
◈ DDIA link — Ch 8 (Unreliable clocks): Snowflake assumes clock monotonicity within each worker. If the system clock goes backwards (NTP correction), you'd generate duplicate IDs. Fix: refuse to generate IDs until clock catches up, or use logical clocks.

URL Shortener (bit.ly)A classic design that covers hashing, redirects, and read-heavy caching

Core flow

POST /shorten {url: "https://example.com/long"} → generate short key (e.g. "aB3kL9") → store: shortKey → longUrl in DB → return: "https://short.ly/aB3kL9" GET /aB3kL9 → check Redis cache first → miss → query DB → return 301 or 302 redirect

301 vs 302: 301 (permanent) — browser caches redirect, fewer server hits. 302 (temporary) — every click hits your server, enabling analytics. Use 302 if you need click tracking.

Generating the short key

  • Hash + truncate: MD5/SHA256 of long URL, take first 7 chars. Problem: collisions → need collision resolution loop
  • Base62 encode a Snowflake ID: generate unique numeric ID, encode in base62 (a-z, A-Z, 0-9). 7 chars = 62⁷ = 3.5 trillion URLs. No collision risk. Cleaner approach

With 100M URLs/day: 100M × 365 × 10 years = 365B URLs. 7-char base62 handles this comfortably.

Scale considerations

  • Read-heavy: reads vastly outnumber writes (every share = many clicks). Cache aggressively in Redis with LRU eviction. 20% of URLs drive 80% of traffic
  • DB: NoSQL (DynamoDB or Cassandra) works well — pure key-value lookups, no joins needed
  • Sharding: shard by hash of short key across DB nodes
  • Analytics: log redirect events to Kafka → stream processing for click counts, geographics

Interview questions

  • Design bit.ly. Walk through the full system for 100M new URLs/day and 10B redirects/day.
  • What's the trade-off between 301 and 302 redirects?
  • How would you handle custom short URLs (vanity URLs) — e.g. bit.ly/stripe-docs?

Web CrawlerDiscovering and indexing the web — a distributed queue and dedup problem

Seed URLs → URL Frontier (priority queue) ↓ Fetcher workers (respect robots.txt, politeness delay) ↓ HTML Parser → extract links ↓ Dedup Filter (Bloom filter: have we seen this URL?) ↓ (new URLs only) URL Frontier ← loop back ↓ (content) Content Store (S3) + Search Index

Key design challenges

  • Scale: web has ~5B pages. Need distributed fetchers across many machines
  • Politeness: don't hammer a single host. Per-domain rate limits, respect robots.txt, crawl-delay headers
  • Priority: crawl PageRank-high pages more often. URL Frontier is a priority queue
  • Deduplication: same content at different URLs. Bloom filter for URL-level; SimHash (fuzzy hash) for near-duplicate content detection
  • Trap avoidance: infinite URL spaces (calendars, session params). Max depth limit, URL normalisation

Bloom filter for dedup important technique

A probabilistic data structure: test URL membership using k hash functions over a bit array. Never false-negatives, tunable false-positive rate. Fits billions of URLs in a few GB of RAM (vs storing every URL in a set).

◈ DDIA link — Ch 3: LSM-tree storage engines use Bloom filters exactly this way — to avoid disk reads for keys that don't exist in an SSTable.

Interview questions

  • How would you detect and avoid crawler traps?
  • How do you deduplicate 10 billion URLs efficiently without storing them all?
  • How would you prioritise which pages to crawl first?

Notification SystemPush, SMS, email at scale — fan-out, retry, and third-party integration

API Server ↓ publish event Message Queue (Kafka) ↓ consumed by workers iOS Push Worker → APNs (Apple) Android Worker → FCM (Google) SMS Worker → Twilio / SNS Email Worker → SendGrid / SES

Design decisions

  • Async via queue: notifications are never in the critical path of the triggering request
  • Retry with backoff: third-party providers fail. Dead-letter queue for repeated failures
  • Rate limiting: protect downstream providers. Also user-level throttling (don't send 50 emails/min to one user)
  • User preferences: DB table: user → [channels, opt-outs, quiet hours]
  • Deduplication: idempotency key per notification event — retry-safe

Reliability concerns

  • APNs and FCM are external — you can't guarantee delivery. Design for at-least-once delivery internally
  • Event log: store every notification attempt + outcome. Enables retry, audit, and support queries
  • Tracking: open/click tracking via pixel or link redirect → Kafka event → analytics DB
◈ DDIA link — Ch 11 (Streams): The Kafka fan-out pattern here is exactly stream processing. Each worker is a consumer group — APNs, FCM, email workers each read the same topic independently.

Interview questions

  • Design a notification system for 100M users. How do you handle failures from APNs/FCM?
  • How do you prevent a user from receiving duplicate notifications after a retry?
  • A user opts out of email but still wants push. How do you model preferences?

News Feed (Facebook/Instagram)Fan-out on write vs fan-out on read — the central architectural choice

Fan-out on write (push model)

When user A posts, immediately push to followers' feed caches. Feed reads are instant — just read pre-computed cache.

  • Pro: reads are O(1), very fast
  • Con: writing is O(followers). Celebrity with 50M followers = 50M cache writes per post. Unacceptable

Fan-out on read (pull model)

When user reads feed, pull recent posts from all followed users and merge-sort them.

  • Pro: write is cheap
  • Con: read is expensive — N DB queries, slow for users following thousands

Hybrid model — what Facebook/Instagram actually do correct answer

Fan-out on write for normal users. Fan-out on read for celebrities (>X followers). At read time, merge the pre-computed feed cache with fresh posts from celebrities you follow.

◈ DDIA link — Ch 1 (Scalability): This is the exact Twitter fan-out problem Kleppmann uses in his scalability chapter. 12 writes/sec (posting) vs 300K reads/sec (timeline fetches). The hybrid model is the solution to that hotspot problem.

Feed ranking & storage

  • Feed cache: Redis sorted set per user, scored by timestamp (or ML ranking score). Trimmed to last 1000 posts
  • Post storage: posts in DB (MySQL/Cassandra), post_id → content. Feed cache only stores post IDs
  • Media: images/videos in CDN + S3. Never in the feed DB
  • Ranking: for chronological feeds, sorted set by timestamp. For algorithmic feeds, ML model scores stored alongside

Interview questions

  • Design Instagram's home feed. How does fan-out work at 500M DAU?
  • How do you handle a celebrity with 100M followers posting? When do you fall back to pull?
  • How would you add algorithmic ranking without slowing down feed reads?

Chat System (WhatsApp / Slack)WebSockets, message ordering, and online presence

Connection layer

  • WebSocket: persistent, bidirectional connection. Server can push messages to client. HTTP polling is an alternative but wasteful
  • Chat servers: stateful — each holds open WebSocket connections. Clients connect to one chat server
  • Service discovery (ZooKeeper): routes new clients to the least-loaded chat server
  • Scale: 1M concurrent WebSockets needs ~10 chat servers (100K connections/server)

Message flow

User A → Chat Server 1 → Message Queue ↓ Chat Server 2 → User B (online) ↓ (if B offline) Push Notification (APNs/FCM)
  • Chat servers don't talk to each other directly — messages route via queue
  • Each message gets a globally unique ID (Snowflake) for ordering
  • Messages stored in DB (HBase/Cassandra) — append-only, read by recency

Message ordering & storage

  • 1-on-1 ordering: sequence number per conversation pair. Easy
  • Group chat ordering: harder. Each message has a Snowflake ID. At read time, merge-sort messages from all senders within the group by ID
  • Storage: KV store (HBase) with key = (channel_id, message_id). Append-only — never update, only insert. This matches Cassandra's write-optimised LSM-tree well
  • Online presence: heartbeat every 5 seconds from client → presence server updates Redis TTL. If key expires → user offline. Fan-out presence changes to friends via pub/sub

Interview questions

  • Why use WebSockets over HTTP long-polling for a chat system?
  • How do you guarantee message ordering in a group chat with 1000 members?
  • How would you implement the "last seen" feature at scale?
  • What happens when a chat server crashes mid-conversation?

Search AutocompleteTrie data structure, top-K results, and low-latency serving

Trie data structure

A tree where each node represents one character. Each prefix path leads to a node that can return suggestions. Store top-K search terms + their frequencies at each node for fast retrieval.

# Trie for ["apple", "app", "apply"] root └─ a └─ p ├─ p [freq=500, top5=["app","apple","apply"...]] │ ├─ l │ │ ├─ e [freq=300] ← "apple" │ │ └─ y [freq=100] ← "apply" └─ ...

System design

  • Data collection: log every search query with frequency. Aggregate via batch job (MapReduce/Spark) daily
  • Trie build: weekly/daily batch job rebuilds trie from aggregated frequencies
  • Serving: trie serialised to Redis or in-memory store on search servers. Sub-millisecond prefix lookups
  • Update cadence: full rebuild weekly; incremental updates more frequently for trending terms
  • Scale: trie sharded by first character or prefix range. Each shard served independently

Interview questions

  • Design Google's search autocomplete. How do you keep suggestions fresh without rebuilding the trie constantly?
  • How would you handle a sudden spike in a trending topic (e.g. a breaking news event)?
  • How do you personalise autocomplete results per user?

YouTube / Video Upload & StreamingBlob storage, transcoding pipelines, and CDN delivery

── Upload flow ───────────────────────────────────────────── Client → API Gateway → Original Storage (S3) ↓ event Transcoding Queue (Kafka) ↓ Transcoding Workers (FFmpeg) → Transcoded Storage (S3) ↓ ↓ Completion Event CDN push / lazy pull ── Playback flow ─────────────────────────────────────────── Client → GeoDNS → CDN Edge Node (cache hit → stream directly) ↓ cache miss S3 Origin → CDN → Client

Video transcoding

  • One upload → many output formats (360p, 720p, 1080p, 4K) and codecs (H.264, VP9, AV1)
  • Directed Acyclic Graph (DAG) of tasks: split → transcode → merge → thumbnail → DRM
  • Parallelism: split video into segments, transcode in parallel across workers
  • Output stored in S3, metadata (title, duration, formats, S3 keys) in DB

Streaming & CDN

  • Adaptive bitrate (ABR): client requests chunks of the right quality for current bandwidth (HLS/DASH). Not one big file download
  • CDN edge caching: popular videos cached at hundreds of PoPs globally. 95%+ cache hit rate for top content
  • Long-tail videos: rare videos served directly from S3 origin (not worth CDN caching)
◈ DDIA link — Ch 10 (Batch) & Ch 11 (Stream): The transcoding pipeline is a batch DAG — same MapReduce/Spark thinking. Segment parallelism = data parallelism in batch processing. The upload completion → transcode trigger is an event-driven stream pattern.

Interview questions

  • Design YouTube. Focus on the upload-to-playback pipeline.
  • How does adaptive bitrate streaming work? Why not just serve one resolution?
  • How would you handle a popular video that gets 10M views in the first minute of upload?

Google Drive / File SyncBlock-level sync, conflict resolution, and metadata vs blob storage

Block-level sync

Don't upload entire files on every change. Split file into fixed-size blocks (e.g. 4MB). Only upload changed blocks.

  • Block identified by hash of its content — deduplication is automatic (same block stored once)
  • File metadata (name, size, block list, order) stored in DB
  • Block content stored in S3
  • Client maintains local block index — computes delta on every file save

Sync flow & conflicts

  • Client → upload new blocks → notify sync service via long-poll or WebSocket
  • Other clients notified → download changed blocks → reconstruct file
  • Conflict: two clients edit same file offline. Last-writer-wins is the simple option. Dropbox: creates "conflicted copy" so neither is lost
  • Offline support: queue local changes. Replay + merge on reconnect
◈ DDIA link — Ch 5 (Multi-leader conflicts): Offline clients are multi-leader. Google Docs uses OT (Operational Transformation). Dropbox uses copy-on-conflict. Same fundamental problem DDIA describes for multi-leader replication.

Interview questions

  • Design Google Drive. How do you minimise bandwidth for large file updates?
  • How do you handle two users editing the same document simultaneously?
  • How would you implement file versioning and restore?

Proximity Service (Yelp / Find Nearby)Geospatial indexing — Geohash vs Quadtree vs R-tree

Geohash most common approach

Encode lat/lng as a short string by recursively dividing the world into a grid. Nearby places share a common prefix.

# Geohash precision levels Length 4: ±20km (city-level) Length 6: ±0.6km (neighbourhood) Length 8: ±19m (building-level) # London example gcpvh → Trafalgar Square area gcpvj → nearby (1 char diff = adjacent cell)

Edge case: two nearby places can be in adjacent cells with different prefixes if they straddle a grid boundary. Fix: search current + 8 surrounding cells.

Quadtree

Recursively subdivide space into 4 quadrants until each contains ≤ N businesses. Tree nodes stored in memory. Good for dynamic data (moving objects).

  • Adapts subdivision to density — cities get finer cells, oceans get coarse ones
  • In-memory tree: fast but requires rebuilding on startup; doesn't scale across servers easily
  • Used by: game location systems, Uber early design

System design for a proximity service

  • Write path: business owner updates location → DB write → invalidate cache → rebuild geohash index
  • Read path: user location → compute geohash at precision 6 → look up Redis sorted set (geohash → business IDs) → fetch business details → sort by distance → return top K
  • Redis GEO commands: Redis has native geospatial support: GEOADD, GEORADIUS, GEODIST. Used internally via a sorted set scored by Geohash
  • Caching strategy: search results for a geohash cell change slowly (businesses don't move often). Cache for minutes

Interview questions

  • Design Yelp's "find restaurants near me". How do you index 200M business locations?
  • Compare Geohash vs Quadtree — when would you choose each?
  • What's the boundary problem in Geohash and how do you fix it?

Nearby Friends (Facebook)Real-time location tracking with WebSockets and pub/sub

Mobile Client → (every 30s) → Location Update Service ↓ Location History DB (Cassandra) ↓ publish Redis Pub/Sub channel per user ↓ subscribers WebSocket Servers → Friends' clients

Key design insight

Don't compute "who is nearby" on every location update. Instead: each user subscribes to location channels of their friends. When a friend publishes a location update, clients compute distance locally. This shifts computation to the client edge.

Scale challenges

  • User with 10,000 friends: 10,000 channel subscriptions. Redis pub/sub can handle this but it's expensive at Facebook-scale
  • Optimisation: only subscribe to friends currently active in the last 10 minutes. Inactive friends pruned from subscription set
  • User with few nearby friends: batched update every 30s is fine. High-activity urban user: more frequent may be needed

Interview questions

  • Design Facebook's "nearby friends" feature. How do you scale to 1B users all sharing location every 30 seconds?
  • How would you reduce battery drain on mobile clients while still keeping location reasonably fresh?

Google MapsRouting, map tiles, and ETA accuracy

Map tile storage & serving

  • World divided into tiles at each zoom level. Z=0 is one tile (whole world). Z=21 is 4 trillion tiles
  • Pre-rendered raster tiles (PNG) stored in S3, cached in CDN
  • Vector tiles (newer): raw geometry sent to client, rendered on device. Smaller, resolution-independent
  • Mobile: download tiles for local area in advance for offline use

Routing (Dijkstra / A*)

  • Road network = weighted graph. Nodes = intersections, edges = road segments with cost (distance, time)
  • Dijkstra: finds shortest path. Too slow for continent-scale graphs
  • A* with heuristic: faster by prioritising paths moving toward destination
  • Contraction Hierarchies (CH): pre-process graph by ranking node "importance". Real maps use CH or similar for sub-second routing at scale
  • Traffic: edge weights updated from real-time GPS probe data (from all drivers). Kafka pipeline ingests → updates routing weights

ETA & live traffic

  • ETA = sum of edge weights on selected route. Weights = historical average + real-time adjustment from current GPS data
  • Machine learning models trained on historical traffic patterns (time of day, day of week, weather, events) → more accurate ETA than naive graph weights
  • Uber/Lyft publish ETA models trained on billions of completed trips

Interview questions

  • How does Google Maps calculate the fastest route across an entire country in under a second?
  • How do you incorporate real-time traffic into routing without recomputing from scratch?
  • How would you handle map tile serving for 1B mobile users?

Distributed Message Queue (Kafka deep dive)Design Kafka from scratch — the Vol 2 treatment

◈ DDIA link — Ch 11 (Stream Processing): This chapter is essentially a first-principles design of Kafka. DDIA explains why the log-based approach is powerful; Alex Xu shows you how to design it. Study them together — DDIA for the "why", this for the "how to discuss it in an interview".

Core design decisions

DecisionChoiceWhy
Storage modelAppend-only log on diskSequential writes are fast even on spinning disk. Immutable — no locking needed for reads
Consumer modelPull-basedConsumer controls its pace. Can batch reads. No push-back pressure problem
Message retentionTime-based (e.g. 7 days)Allows replay, reprocessing, multiple independent consumers
DurabilityReplication factor 3 + ISRLeader + 2 replicas. Producer waits for ISR ack before confirming
OrderingWithin partition onlyCross-partition ordering impossible without coordination — by design
Consumer trackingConsumer commits offsetKafka doesn't track per-consumer state. Decoupled — broker doesn't need to know consumer progress

Producer batching & compression

  • Producers batch messages before sending (configurable linger.ms)
  • Compression (Snappy, LZ4, zstd) applied to batches — significant bandwidth saving
  • Idempotent producer: each message has a sequence number — broker deduplicates retries
  • Transactional producer: write to multiple partitions atomically — basis for exactly-once

Consumer group rebalancing

  • Partitions distributed across consumers in a group. One partition → one consumer (within group)
  • Consumer joins/leaves → rebalance triggered by Group Coordinator (a broker)
  • Eager rebalance: all consumers stop, re-assign all partitions. Causes pause
  • Cooperative rebalance: only affected partitions reassigned. Minimal disruption. Kafka 2.4+

Interview questions

  • Design a distributed message queue. What are the core trade-offs between Kafka and RabbitMQ?
  • How does Kafka achieve exactly-once delivery? What are the components involved?
  • A consumer is processing too slowly and falling behind. How do you diagnose and fix consumer lag?
  • Why does Kafka use pull-based consumption rather than push?

Metrics Monitoring & Alerting (Datadog / Prometheus)Time-series data at scale with fast aggregation queries

Data model — time series

# A metric is: name + labels + value + timestamp http_requests_total{ service="payments", status="200", region="eu-west" } 1234567 @ 1718000000

High cardinality labels (user_id, request_id) are a common mistake — they create millions of unique time series and overwhelm the storage layer.

Storage — why TSDB not SQL

  • Write pattern: append-only, high rate (millions of data points/sec)
  • Read pattern: time-range scans with aggregations (avg, p99, sum)
  • Column-oriented storage per metric: all timestamps together, all values together → excellent compression (delta encoding + run-length)
  • Prometheus: custom TSDB. Chunks of 2-hour blocks. Old chunks compacted
  • InfluxDB, TimescaleDB, ClickHouse: alternatives with SQL-like query

Collection & alerting pipeline

Services / hosts ↓ scrape (Prometheus) or push (StatsD/Telegraf) Metrics Collector → Kafka (buffer, decouple at scale) ↓ Time-Series DB (Prometheus / InfluxDB) ↓ ↓ Query Layer Alert Evaluator (Grafana) (check rules every N sec) ↓ threshold breach PagerDuty / Slack
◈ DDIA link — Ch 3 (OLAP) & Ch 10 (Batch): TSDB is a specialised columnar store — same columnar compression principles DDIA describes for OLAP. The metrics pipeline is also a mini Lambda architecture: real-time stream for alerting, batch rollup for long-term storage.

Ad Click Event AggregationHigh-volume stream aggregation, exactly-once counting, and fault tolerance

Pipeline design

Ad clicks (billions/day) ↓ Kafka topic: ad-clicks (partitioned by ad_id) ↓ Flink aggregation job - tumbling window: count clicks per ad per minute - watermark handles late-arriving events (up to 5 min late) - checkpoint state to S3 for fault tolerance ↓ Result store: ClickHouse / Cassandra key=(ad_id, minute_bucket), value=click_count ↓ Query API → Advertisers dashboard

Exactly-once counting

  • Clicks are money — double-counting or losing counts has direct financial impact
  • Dedup at ingest: each click event has a unique ID. Dedup via Redis set with TTL before inserting to Kafka
  • Flink exactly-once: two-phase commit from Flink to sink. Flink checkpoints Kafka offsets atomically with output state
  • Reconciliation job: daily batch recount from raw log (source of truth) vs streaming aggregate. Reconcile discrepancies

Late data handling

  • Mobile clicks arrive late (app was offline). Can't just drop them
  • Watermark: Flink tracks "how far behind is the slowest event stream"
  • Hold windows open for N minutes past their end time before closing
  • After window closes: late events trigger result correction. Advertiser sees updated counts
◈ DDIA link — Ch 11 (Streaming): This is the worked example for everything DDIA says about windowing, watermarks, and exactly-once stream processing. If you understand this design, you understand Ch 11 applied in practice.

Hotel Reservation (Booking.com)Inventory management, double-booking prevention, concurrency

The double-booking problem

Two users book the last room simultaneously. Both see "available". Both complete checkout. One booking must fail — but which one and when?

  • Optimistic locking: UPDATE rooms SET booked=true WHERE room_id=X AND booked=false. Check rows affected = 1. If 0, someone else got it — show error to user
  • Pessimistic locking: SELECT ... FOR UPDATE. Serialises access. Safe but slow under high concurrency
  • Database constraint: unique constraint on (room_id, date) in bookings table. DB enforces — double-insert fails with constraint violation

Inventory model

# Approach: track remaining inventory per room per date room_inventory table: hotel_id | room_type | date | total | reserved 101 | double | 2025-08-15 | 10 | 9 # Booking = decrement reserved count atomically UPDATE room_inventory SET reserved = reserved + 1 WHERE hotel_id=101 AND room_type='double' AND date='2025-08-15' AND reserved < total ← prevents overbooking

Distributed concurrency at scale

  • Caching inventory: cache available count in Redis for fast display. But cache can lag — always do final authoritative check against DB at booking time
  • Idempotency key: client sends a booking_token with each attempt. DB unique constraint on booking_token prevents double-charge if user retries
  • Distributed lock (Redlock): lock on (room_id, date) in Redis for the duration of the booking transaction. Prevents concurrent bookings for same room. Risk: Redlock has known correctness issues under clock skew
◈ DDIA link — Ch 7 (Transactions): This is write skew in practice. Two transactions both read "room available", both decide to book, both write — result violates the invariant "room can't be booked twice". Solutions: serializable isolation, SELECT FOR UPDATE, or DB unique constraint (the most practical).

S3-like Object StoragePetabyte-scale blob storage — data plane vs control plane

── Control plane ────────────────────────────────────────────── IAM (auth) Metadata Service → RocksDB / strong consistent store bucket → [object_key → {size, checksum, location, version}] ── Data plane ───────────────────────────────────────────────── API Gateway ↓ PUT /bucket/key Data Service → split into chunks → write to 3 data nodes (erasure coding) ↓ on success → update metadata Metadata: key=object_id, location=[node1, node2, node3] ── Durability ───────────────────────────────────────────────── Erasure coding (e.g. 8+4 Reed-Solomon): 12 chunks, any 8 can reconstruct Cheaper than 3x replication (~1.5x overhead vs 3x)

Key design decisions

  • Flat namespace: S3 has no real directories — "folders" are just key prefixes. Simplifies metadata
  • Multipart upload: large files split and uploaded in parallel chunks. Each part individually checksummed. Enables resumable uploads
  • Versioning: each PUT creates a new version. Delete is a "delete marker" — old versions still retrievable
  • Checksums: SHA-256 per chunk, verified on write and periodically during scrubbing to detect bit rot

Metadata storage

  • Bucket listing requires scanning all keys with a prefix — expensive at scale
  • Metadata must be strongly consistent (S3 is strongly consistent since 2020)
  • RocksDB or a distributed KV store on SSDs. Each key = object path, value = metadata JSON
  • Separate from data plane — can scale independently. Metadata is tiny compared to data

  • What is erasure coding and why is it better than 3x replication for large files?
  • How does S3 achieve 11 nines of durability?

Payment SystemExactly-once money movement, idempotency, and reconciliation — most relevant for you

⚠️ This is the most directly relevant design for Stripe, Wise, and Monzo interviews. Master every component here. They will probe your understanding of consistency, idempotency, and failure handling in a financial context.
Client ↓ POST /payments {amount, currency, idempotency_key} API Gateway (auth, rate limit) ↓ Payment Service 1. Check idempotency_key → already exists? Return cached response 2. Validate: funds available? limits OK? 3. Write payment record (PENDING) to DB 4. Call PSP (Stripe/Adyen) → external network call — may fail 5. PSP responds SUCCESS/FAIL 6. Update payment record → COMPLETED / FAILED 7. Publish event to Kafka → downstream systems (ledger, notifications) ↓ Client ← response

Idempotency — the central pattern interview essential

  • Client sends a unique idempotency_key with every payment request (UUID)
  • Server stores (idempotency_key → response) in DB before calling PSP
  • On retry: check DB first → if found, return stored response without re-processing
  • This makes payment creation safe to retry unconditionally
  • Stripe does exactly this — their API docs require idempotency keys for all POST requests

Handling PSP failures

  • PSP call times out — you don't know if it succeeded
  • Query PSP for status (poll or webhook callback) before making any retry decision
  • Don't retry blindly — PSP may have already charged the card
  • Store payment in PENDING state. Background job reconciles PENDING payments older than X minutes against PSP
  • This is the fundamental exactly-once problem in payments

Double-entry ledger

  • Every payment creates two ledger entries: debit (money out) + credit (money in). They must always balance
  • Append-only immutable ledger — never update or delete rows. Corrections made by new entries
  • Balance = sum of all credits - sum of all debits for an account. Computed from ledger, not stored as a field (avoids inconsistency)
  • Reconciliation: nightly batch job sums ledger, compares with PSP settlement reports. Any discrepancy triggers investigation
  • This is how all banks and fintechs (Monzo, Wise, Stripe) structure their accounting data
◈ DDIA link — Ch 12 (Future of Data Systems): The immutable append-only event log Kleppmann advocates is exactly this ledger pattern. Financial systems figured this out decades ago. Kleppmann argues all data systems should work this way.

  • What is a double-entry ledger and why do payment systems use it?

Digital WalletBalance accuracy at scale — distributed transactions without 2PC

The core challenge: transferring funds across two accounts

# Naive approach — WRONG under distributed failures UPDATE accounts SET balance = balance - 100 WHERE id = 'alice'; UPDATE accounts SET balance = balance + 100 WHERE id = 'bob'; # If second UPDATE fails: alice loses $100, bob gets nothing → money disappears # Correct: wrap in a transaction (single DB) BEGIN; UPDATE accounts SET balance = balance - 100 WHERE id = 'alice' AND balance >= 100; UPDATE accounts SET balance = balance + 100 WHERE id = 'bob'; COMMIT;

When accounts are on different shards

  • Alice on Shard 1, Bob on Shard 2 — can't use a local transaction
  • Option 1 — 2PC: works but coordinator failure risk. Viable for low-throughput financial systems where correctness trumps everything
  • Option 2 — Saga: debit Alice (local tx) → event → credit Bob (local tx). If credit fails: compensate by crediting Alice back. Eventual consistency
  • Option 3 — Colocate: shard accounts so payer + payee are on the same shard. Only works if you can predict pairs (hard for P2P wallets)

Balance read consistency

  • Reading balance from a replica can return stale data — user sees wrong available balance
  • For balance reads: always read from leader (or use linearizable read protocol)
  • Balance check before debit must be in the same transaction as the debit — never check then act in separate steps
◈ DDIA link — Ch 7 (Transactions) & Ch 9 (Consistency): The cross-shard transfer is the distributed transaction problem from Ch 7 (2PC, Sagas). The balance read requirement is the linearizability requirement from Ch 9.

Stock ExchangeMatching engine, order book, and microsecond latency

Order book & matching engine

  • Order book: two sorted lists — bids (buy orders, price DESC) + asks (sell orders, price ASC)
  • Match: top bid ≥ lowest ask → trade executes at the ask price
  • In-memory: matching engine runs entirely in RAM for speed. Single-threaded to avoid locking (like Redis). Deterministic
  • Data structure: price-level doubly linked list. O(1) best bid/ask. O(1) order add/cancel

Latency requirements

  • Co-location: trading firms pay to physically place servers next to exchange servers. Speed of light is the bottleneck
  • Kernel bypass (DPDK): avoid OS network stack overhead
  • Lock-free data structures: avoid mutex contention
  • No GC: Java GC pauses are fatal. C++ or GC-tuned Java with off-heap storage
  • Target: nanosecond to microsecond order processing

Sequencer & fault tolerance

  • Sequencer: assigns a monotonically increasing sequence number to every order. This is the global order of events. All replicas must process events in sequence order → deterministic matching
  • Event sourcing: order book state is derived from replaying the sequence of order events. Crash recovery = replay from checkpoint + recent events (same as Flink checkpointing)
  • Hot standby: primary matching engine + hot standby replays same events. On failure: standby takes over from its last processed sequence number
◈ DDIA link — Ch 11 (Streams) & Ch 9 (Consensus): The sequencer is a total order broadcast — the same primitive underlying Raft log replication. The event-sourced order book is the DDIA concept of deriving state from an immutable event log.
